## [Dev Log] 데이터의 흐름으로 기록하는 성장: 대시보드 자동화와 기록의 가치

안녕하세요, 씨피랩스의 CTO 송주한(Jeffrey)입니다.

엔지니어에게 있어 '기록'은 단순한 메모 이상의 의미를 갖습니다. 내가 어떤 고민을 했고, 어떤 기술적 결정을 내렸으며, 시간이 흐름에 따라 어떻게 성장했는지를 보여주는 가장 객관적인 지표가 되기 때문입니다. 하지만 매일 수많은 커밋을 남기면서 이를 일일이 수동으로 정리하는 것은 매우 비효율적인 일이죠.

오늘은 제가 개인적으로 운영하고 있는 프로젝트 대시보드의 데이터를 업데이트하며, '자동화된 기록'이 주는 인사이트에 대해 이야기해보려 합니다.

### 🛠 오늘의 작업: 대시보드 데이터 동기화

오늘 진행한 작업은 단순해 보이지만, 제 개발 생태계를 한눈에 파악할 수 있게 해주는 `dashboard.json` 파일의 업데이트였습니다. 

```json
// src/data/dashboard.json 변경 사항 일부
- "last_msg": "feat: Add project dashboard page + fix MDX angle bracket issue",
+ "last_msg": "Auto-generate weekly recap",
...
+ {
+   "name": "trump-code",
+   "last_date": "2026-04-11",
+   "last_msg": "chore: daily pipeline data update 2026-04-11",
+   "commits_30d": 471
+ }
```

이번 업데이트에서 주목할 점은 단순히 날짜를 바꾸는 것이 아니라, 여러 레포지토리의 활동성(Activity)과 커밋 빈도를 정량적으로 추적하고 있다는 점입니다. 특히 `trump-code` 레포지토리의 경우, 최근 30일간 471건의 커밋이 기록된 것을 볼 수 있습니다. 이는 단순한 숫자가 아니라, 해당 기간 동안 진행된 파이프라인 고도화와 데이터 업데이트의 치열함을 증명하는 수치입니다.

### 💡 기술적 결정과 인사이트: 왜 '데이터화' 하는가?

저는 블록체인, Web3, AI와 같은 최첨단 기술 분야에서 20년 넘게 커리어를 쌓아오며 한 가지 깨달은 점이 있습니다. **"측정할 수 없는 것은 개선할 수 없다"**는 것입니다.

많은 개발자가 깃허브의 '잔디 심기'에 집중하지만, 저는 여기서 한 단계 더 나아가 **어떤 프로젝트에 내 에너지가 집중되고 있는지**를 데이터로 시각화하는 방식을 택했습니다. `activity` 배열을 통해 요일별 활동량을 추적하고, `commits_7d`와 `commits_30d`를 통해 단기/장기 추세를 분석합니다.

이러한 방식은 다음과 같은 이점을 줍니다:
1. **리소스 배분의 최적화**: 특정 프로젝트에 과하게 매몰되어 있지는 않은지, 혹은 방치된 프로젝트는 없는지 객관적으로 파악할 수 있습니다.
2. **성취감의 가시화**: `Auto-generate weekly recap`과 같은 자동화 로직을 통해, 내가 만든 시스템이 나를 위해 기록을 남겨주는 '선순환 구조'를 구축했을 때의 쾌감은 개발자만이 느낄 수 있는 즐거움입니다.
3. **히스토리 추적**: 나중에 특정 시점에 내가 어떤 기술적 고민을 했는지 `last_msg`를 통해 빠르게 역추적할 수 있습니다.

### 마치며

씨피랩스에서 수많은 복잡한 아키텍처를 설계하고 팀을 리딩하고 있지만, 결국 가장 기본이 되는 것은 '기본에 충실한 기록'이라고 생각합니다. 거창한 문서화가 아니더라도, 오늘처럼 JSON 파일 하나에 나의 발자취를 남기는 습관이 모여 거대한 기술적 자산이 됩니다.

여러분은 자신의 성장을 어떻게 기록하고 계신가요? 단순한 기록을 넘어, 그것을 어떻게 '데이터화'하여 인사이트를 얻을 수 있을지 고민해 보시는 것을 추천드립니다.
