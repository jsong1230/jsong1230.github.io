# [Dev Log] 라디오의 꽃, '드라마'와 '가변 편성'으로 완성하는 Heritage Radio Phase 2

안녕하세요, 씨피랩스의 CTO 송주한(Jeffrey Joo-Han Song)입니다.

최근 진행 중인 'Heritage Radio' 프로젝트가 Phase 1 MVP를 넘어, 서비스의 깊이를 더하는 **Phase 2** 단계에 진입했습니다. 단순한 기능 구현을 넘어 '라디오라는 매체의 본질'을 어떻게 기술적으로 재현할 것인가에 집중한 이틀간의 치열한 작업 기록을 공유합니다.

### 1. 문제 정의: "고정된 슬롯에서 살아있는 편성표로"

Phase 1에서는 13개의 고정된 시간 슬롯을 사용했습니다. 하지만 실제 라디오는 요일별로 프로그램이 다르고, 특정 요일에만 방영되는 특집 드라마가 있는 등 매우 유동적입니다. 특히 "라디오의 꽃은 드라마"라는 요구사항을 충족시키기 위해, 기존의 정적인 구조를 완전히 갈아엎어야 했습니다.

**핵심 도전 과제:**
- 13개 고정 슬롯 $\rightarrow$ 요일별 가변 21슬롯 체계로 전환
- 드라마 5종(새벽 소설, 일일 연속극 등)의 특수 편성 반영
- 12가지 콘텐츠 분류(Drama, Pop, Trot, Yasa 등)에 따른 맞춤형 생성 로직 구축

### 2. 기술적 결정과 구현

#### 가변 편성 아키텍처와 DB 스키마 확장
단순히 데이터를 추가하는 것이 아니라, `programs_v2`와 `weekday_schedule_v2` 테이블을 새롭게 설계하여 요일별/시간별 매핑 구조를 구축했습니다. 이를 통해 월~금 21개, 토 19개, 일 18개라는 복잡한 편성표를 유연하게 관리할 수 있게 되었습니다.

#### AI 에이전트의 '페르소나' 세분화
콘텐츠 타입이 12종으로 늘어남에 따라, Gemma4 프롬프트를 각 타입에 맞게 최적화했습니다. 특히 흥미로웠던 점은 **`time_hint`** 도입입니다. 아침 프로그램에서 "편안한 밤 되세요"라고 말하는 AI의 실수를 방지하기 위해, 생성 시점의 시간대 정보를 프롬프트에 주입하여 자연스러운 인사말을 유도했습니다.

#### 음원 서비스의 Fallback 전략
YouTube Premium 계정 연동을 통한 API 검색을 기본으로 하되, 연결되지 않은 환경에서도 끊김 없는 경험을 제공하기 위해 **Suno AI 기반의 Fallback 시스템**을 구축했습니다. 
`slot_id`의 해시값을 이용해 14곡의 풀에서 곡을 선택하게 함으로써, 사용자가 매번 같은 곡을 듣지 않고 슬롯별로 다른 음악을 경험하도록 설계했습니다.

### 3. 코드에서 배운 인사이트

이번 작업 중 가장 신경 쓴 부분은 효율적인 배치(Batch) 처리였습니다. 142편에 달하는 에피소드를 생성하면서 이미 생성된 데이터를 중복 처리하지 않도록 하는 로직이 중요했습니다.

```python
# episode_generator_v2.py 의 핵심 변경점
# 기존 ready 상태인 에피소드는 건너뛰어 리소스 낭비 방지
if episode.status == 'ready':
    continue 
```

단순한 `if` 문 하나지만, 대량의 AI API 호출이 발생하는 환경에서는 비용과 시간을 결정짓는 결정적인 최적화 포인트가 됩니다.

### 4. 마치며: 기술과 감성의 접점

프론트엔드에서는 `Noto Serif KR` 폰트와 3% 투명도의 빈티지 노이즈 텍스처를 적용하여 60-80년대 라디오의 아날로그 감성을 재현했습니다. 

엔지니어로서 저는 종종 '기능'에 매몰되곤 합니다. 하지만 이번 Phase 2 작업을 통해, **최신 AI 기술(Gemma4, Suno AI)이 아날로그의 향수라는 감성과 만났을 때 얼마나 강력한 사용자 경험을 만들어낼 수 있는지** 다시 한번 깨달았습니다.

이제 자동 이어재생과 Spotify 연동이라는 마지막 퍼즐 조각만 남겨두고 있습니다. 더 완성도 높은 '시간 여행 라디오'로 돌아오겠습니다.
