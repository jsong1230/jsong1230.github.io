# [Dev Log] 데이터의 흐름으로 기록하는 성장: 대시보드 업데이트와 그 의미

안녕하세요, 씨피랩스의 CTO 송주한(Jeffrey Joo-Han Song)입니다.

엔지니어로서 20년 넘게 코드를 작성하며 제가 깨달은 것 중 하나는, **'측정되지 않는 것은 개선될 수 없다'**는 점입니다. 단순히 기능을 구현하는 것을 넘어, 내가 어디에 시간을 쓰고 있는지, 어떤 프로젝트가 활발하게 움직이고 있는지를 시각화하는 것은 개인의 생산성 관리뿐만 아니라 기술적 방향성을 잡는 데 매우 중요합니다.

오늘은 제 개인 대시보드의 데이터를 업데이트하며, 최근의 작업 흐름을 되짚어보는 시간을 가졌습니다.

### 오늘의 작업: 대시보드 데이터 갱신

오늘 진행한 작업은 간단해 보이지만 중요한 의미를 갖는 `chore: Update dashboard data`였습니다. `src/data/dashboard.json` 파일을 업데이트하여 최근 7일 및 30일간의 커밋 활동량과 최신 커밋 메시지를 반영했습니다.

```json
// 변경 전
"generated_at": "2026-06-23T23:00:01.813Z",
// 변경 후
"generated_at": "2026-06-24T23:00:01.630Z",
```

단순한 날짜 변경 같지만, 이 JSON 데이터는 제 활동의 '지문'과 같습니다. 특히 이번 업데이트에서 흥미로운 점은 각 레포지토리의 `activity` 배열 변화였습니다.

### 기술적 인사이트: 데이터의 '추세'를 읽는 법

이번 diff에서 눈에 띄는 부분은 `metadium_homepage`나 `heritageradio` 같은 프로젝트들의 활동 주기 변화입니다.

```diff
- "commits_7d": 95,
+ "commits_7d": 43,
```

특정 시점에 폭발적으로 증가했던 커밋 수가 다시 안정화되는 과정을 보며, 프로젝트의 생명주기(Lifecycle)를 체감합니다. 초기 셋업이나 대규모 리팩토링 단계에서는 커밋 수가 급증하지만, 이후 유지보수 및 최적화 단계로 접어들면 커밋의 '양'보다는 '질'과 '정교함'이 중요해집니다.

특히 `metadium_homepage` 작업 중 **"cplabs 잔재 정리"**라는 메시지를 남겼던 부분을 다시 보게 되었습니다. 씨피랩스로의 브랜드 전환 과정에서 과거의 흔적을 지우고 정체성을 확립하는 작업은 단순한 텍스트 변경이 아니라, 시스템 전체의 일관성을 맞추는 중요한 엔지니어링 과정이었습니다.

### 마치며: 기록이 주는 힘

많은 분이 "단순한 데이터 업데이트에 왜 이렇게 공을 들이느냐"고 물으실지도 모릅니다. 하지만 저는 이렇게 정량화된 데이터를 통해 제 상태를 객관적으로 바라봅니다.

- **과몰입 방지:** 특정 프로젝트에만 너무 치우쳐 있지는 않은가?
- **방치된 프로젝트 발견:** 오랫동안 업데이트되지 않은 레포지토리는 왜 멈춰 있는가?
- **성취감의 시각화:** 꾸준히 쌓인 `commits_30d` 수치를 보며 얻는 동기부여.

Web3, AI, DID와 같이 빠르게 변하는 기술 스택을 다루는 전문가일수록, 중심을 잡기 위한 '자기 객관화'가 필요합니다. 저에게 이 대시보드는 단순한 숫자의 나열이 아니라, 매일 조금씩 성장하고 있는 엔지니어로서의 기록지입니다.

내일은 또 어떤 데이터가 쌓일지 기대하며, 오늘의 로그를 마칩니다.
