## [Dev Log] 데이터로 기록하는 성장의 궤적: 대시보드 업데이트와 그 너머의 인사이트

안녕하세요, 씨피랩스의 CTO 송주한(Jeffrey)입니다.

엔지니어에게 있어 '기록'은 단순한 메모 이상의 의미를 갖습니다. 제가 작성하는 코드가 서비스의 기능을 결정한다면, 제가 남기는 기록은 그 기능이 왜 필요했는지, 그리고 어떤 고민을 통해 구현되었는지를 증명하는 이정표가 되기 때문입니다.

오늘은 개인 대시보드(`jsong1230.github.io`)의 데이터를 업데이트하며, 최근 제가 집중하고 있는 기술적 관심사와 작업 흐름을 정리해 보았습니다. 단순한 `json` 파일의 변경처럼 보이지만, 그 안에는 현재 제가 풀고 있는 문제들의 단면이 담겨 있습니다.

### 1. 데이터 업데이트: 정적인 기록에서 동적인 흐름으로

이번 작업의 핵심은 `src/data/dashboard.json` 파일의 갱신이었습니다. 저는 제가 관리하는 여러 리포지토리의 활동량(commits)과 최신 커밋 메시지를 시각화하여 관리합니다. 

이번 diff에서 흥미로운 점은 단순한 숫자 업데이트가 아니라, **현재 진행 중인 프로젝트들의 성격 변화**가 드러난다는 점입니다.

```json
{
  "name": "mirae",
  "last_date": "2026-07-03",
  "last_msg": "feat: AI 캐펙스 조기경보 canary_alert.sh (코어위브·오라클 감시)",
  "branch": "main",
  ...
}
```

특히 위 부분에서 볼 수 있듯이, 최근 AI 인프라 비용(CAPEX) 관리와 모니터링에 집중하고 있습니다. CoreWeave나 Oracle 같은 GPU 클라우드 환경에서 자원 낭비를 막기 위한 '조기 경보 시스템'을 구축하는 작업인데, 이는 AI 모델의 성능만큼이나 '비용 효율적인 운영'이 Web3와 AI 융합 서비스의 핵심 경쟁력이 되기 때문입니다.

### 2. 기술적 결정: 자동화와 가시성의 조화

저는 반복적인 데이터 업데이트를 수동으로 하지 않습니다. `Auto-generate daily posts`라는 커밋 메시지에서 알 수 있듯이, 자신의 활동 데이터를 자동으로 수집하고 반영하는 파이프라인을 구축해 두었습니다.

여기서 제가 내린 기술적 결정은 **"최소한의 오버헤드로 최대의 가시성을 확보하는 것"**입니다. 복잡한 DB를 구축하는 대신, 정적 사이트 생성기(SSG) 환경에 맞게 JSON 파일을 업데이트하고 Git 커밋을 통해 버전 관리를 하는 방식을 택했습니다. 이는 데이터의 변경 이력을 그대로 Git 히스토리로 남길 수 있어, 나중에 "내가 이때 어떤 고민을 했지?"라고 되짚어볼 때 매우 유용합니다.

### 3. 이번 작업을 통해 얻은 인사이트

코드의 변경 사항 중 `metadium_homepage` 리포지토리의 변경 건이 눈에 띕니다.

> `"last_msg": "monitor.sh에 Telegram 알림 추가 (이상 감지 시 push)"`

단순히 모니터링 스크립트를 짜는 것에 그치지 않고, 이를 '알림'이라는 액션으로 연결하는 과정입니다. 엔지니어로서 가장 경계해야 할 것은 **"모니터링은 하고 있지만, 아무도 보지 않는 상태"**입니다. 장애가 발생했을 때 시스템이 스스로 나에게 말을 걸게 만드는 구조, 즉 'Push형 알림' 체계를 구축하는 것이 운영 안정성을 비약적으로 높이는 길임을 다시 한번 확인했습니다.

### 마치며

20년 넘게 엔지니어와 CTO로 활동하며 느낀 점은, 결국 좋은 소프트웨어는 '세밀한 관찰'과 '끈질긴 자동화'에서 나온다는 것입니다. 

오늘의 대시보드 업데이트는 단순한 수치 변경이었지만, 이를 통해 제가 어디에 시간을 쓰고 있고 어떤 문제를 해결하고 있는지 객관적으로 바라볼 수 있었습니다. 씨피랩스에서도 이러한 데이터 기반의 의사결정과 자동화 문화를 통해 더 견고한 기술 생태계를 만들어가고자 합니다.

오늘 여러분의 대시보드에는 어떤 기록이 남았나요?
