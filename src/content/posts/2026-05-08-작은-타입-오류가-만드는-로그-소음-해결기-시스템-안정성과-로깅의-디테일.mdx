# [Dev Log] 작은 타입 오류가 남기는 큰 소음: 로깅의 디테일과 시스템 안정성

안녕하세요, 씨피랩스의 CTO 송주한(Jeffrey Joo-Han Song)입니다.

엔지니어로 오래 일하다 보면, 시스템의 핵심 로직은 완벽하게 돌아가고 있는데 정작 '로그' 때문에 시스템이 지저분해지는 경험을 종종 하게 됩니다. 오늘은 아주 사소해 보이지만, 운영 관점에서는 꽤나 신경 쓰였던 타입 캐스팅 이슈를 해결한 과정을 공유해보려 합니다.

### 문제의 발단: "작동은 하는데, 로그가 시끄럽다"

최근 `qwen3tts`를 이용해 대규모 오디오 배치 합성 작업을 진행했습니다. 오디오 처리 파이프라인에서는 소리 크기를 일정하게 맞추는 '정규화(Normalization)' 과정이 필수적인데, 저는 `ffmpeg`의 `loudnorm` 필터를 사용해 이를 처리하고 있습니다.

문제는 정규화 작업 자체는 성공적으로 수행되어 결과물은 완벽했지만, 작업 완료 후 남겨지는 로그에서 `TypeError`가 발생하며 스택 트레이스(Stack Trace)가 계속해서 찍히고 있었다는 점입니다.

### 기술적 분석과 결정

문제가 된 코드는 다음과 같았습니다.

```python
# 변경 전
logger.info(
    "loudnorm OK %s — before I=%.2f LUFS · TP=%.2f → -16 LUFS 통일",
    p.name, before["input_i"], before["input_tp"],
)
```

`ffmpeg`의 `loudnorm` 결과값이 담긴 JSON을 파싱했을 때, 통합 강도(`input_i`)와 트루 피크(`input_tp`) 값이 숫자형이 아닌 **문자열(string)**로 들어오고 있었습니다. 파이썬의 `%.2f` 포맷팅은 엄격하게 float 타입을 기대하기 때문에, 문자열이 들어오는 순간 `TypeError`를 던지며 로그 출력이 실패한 것이죠.

사실 비즈니스 로직상으로는 아무런 문제가 없었습니다. 정규화는 이미 끝났고, 파일도 정상적으로 저장되었습니다. 하지만 수백, 수천 개의 파일을 배치 처리하는 상황에서 매번 에러 로그가 남는다는 것은 운영상 매우 위험한 신호입니다. **진짜 심각한 에러가 발생했을 때, 이 '가짜 에러'들에 묻혀 발견하지 못할 가능성(Alert Fatigue)**이 크기 때문입니다.

결국 해결책은 단순했습니다. 명시적으로 `float()` 캐스팅을 추가하는 것이었습니다.

```python
# 변경 후
logger.info(
    "loudnorm OK %s — before I=%.2f LUFS · TP=%.2f → -16 LUFS 통일",
    p.name, float(before["input_i"]), float(before["input_tp"]),
)
```

### 이번 작업에서 얻은 인사이트

이번 수정은 단 한 줄의 코드 변경이었지만, 두 가지 중요한 교훈을 다시금 생각하게 했습니다.

**첫째, 외부 라이브러리/도구의 출력값은 절대 신뢰하지 마세요.**
`ffmpeg` 같은 강력한 도구라도 JSON 출력 형식이 버전이나 환경에 따라 미묘하게 다를 수 있습니다. 외부 데이터가 시스템 내부로 들어오는 접점(Boundary)에서는 항상 명시적인 타입 검증이나 캐스팅을 거치는 것이 안전합니다.

**둘째, '깨끗한 로그'는 그 자체로 시스템의 품질입니다.**
작동만 한다고 해서 끝이 아닙니다. 불필요한 노이즈 로그를 제거하는 것은 단순히 보기 좋게 만드는 작업이 아니라, 장애 대응 시간을 단축시키고 시스템의 가시성을 확보하는 중요한 엔지니어링 활동입니다.

씨피랩스에서 추구하는 기술적 지향점 역시 이와 같습니다. 화려한 기능 구현보다 더 중요한 것은, 보이지 않는 곳에서 정교하게 다듬어진 안정성입니다.

오늘도 작은 디테일을 챙기며 더 단단한 시스템을 만들어갑니다.
