# [Dev Log] 데이터의 흐름으로 기록하는 나의 성장 곡선: Dashboard 업데이트

안녕하세요, 송주한(Jeffrey Joo-Han Song)입니다.

엔지니어로서 20년 넘게 수많은 프로젝트를 거쳐오며 제가 깨달은 것 중 하나는, **'측정되지 않는 것은 개선될 수 없다'**는 사실입니다. 코드의 퀄리티나 아키텍처의 정교함도 중요하지만, 내가 어디에 시간을 쓰고 있고 어떤 리듬으로 개발하고 있는지를 시각화하는 것은 장기적인 생산성 관리에 매우 큰 도움이 됩니다.

오늘은 제 개인 대시보드(`jsong1230.github.io`)의 데이터를 업데이트하며, 최근의 작업 흐름을 되돌아보는 시간을 가졌습니다.

### 🛠️ 오늘의 작업: 대시보드 데이터 동기화

오늘 진행한 작업은 단순해 보이지만, 제 개발 생태계를 한눈에 보여주는 `dashboard.json` 파일을 최신화하는 것이었습니다. 단순히 숫자만 바꾼 것이 아니라, 현재 제가 집중하고 있는 프로젝트들의 우선순위와 활동량이 어떻게 변하고 있는지를 데이터로 확인하는 과정이었습니다.

#### 기술적 포인트: 데이터 기반의 활동 추적
이번 diff에서 흥미로운 부분은 각 레포지토리의 `activity` 배열과 `commits` 통계의 변화입니다.

```json
// heritageradio 레포지토리의 변화
- "commits_7d": 8,
- "commits_30d": 8
+ "commits_7d": 14,
+ "commits_30d": 41
```

`heritageradio` 프로젝트의 활동량이 급증한 것을 볼 수 있습니다. 특히 `[landing→main] 7월 계획 doc 정정 — 110 DB 실측 대조`라는 커밋 메시지에서 알 수 있듯, 단순한 기능 구현을 넘어 실제 DB 실측 데이터와 대조하며 설계를 정교화하는 단계에 진입했음을 알 수 있습니다. 

또한, 씨피랩스에서의 경험을 바탕으로 진행 중인 `metadium_homepage` 작업에서는 "cplabs 잔재 정리"와 같은 리팩토링 작업이 진행되었습니다. 기술적 부채를 털어내고 시스템을 깨끗하게 유지하는 것은 CTO로서 제가 항상 강조하는 '지속 가능한 개발'의 핵심입니다.

### 💡 인사이트: '커밋 로그'라는 이름의 이정표

많은 개발자가 커밋 메시지를 단순히 "작업 완료"나 "수정" 정도로 남기곤 합니다. 하지만 저는 커밋 로그를 미래의 나에게 보내는 편지라고 생각합니다. 

오늘 업데이트한 데이터 속의 `last_msg`들을 보면, 단순히 '무엇을 했다'가 아니라 '왜 했는지', 그리고 '어떤 기준(DB 실측 대조 등)'으로 작업했는지가 드러나 있습니다. 이렇게 기록된 데이터가 쌓여 대시보드에 시각화될 때, 저는 제가 단순히 코드를 치는 기계가 아니라 **문제를 해결해 나가는 과정**에 있다는 확신을 얻습니다.

### 마치며

블록체인, Web3, AI와 같은 최첨단 기술을 다루는 전문가일수록 기본으로 돌아가는 습관이 중요합니다. 그 기본이란 바로 **'기록'과 '측정'**입니다.

오늘의 데이터 업데이트를 통해 제가 현재 어떤 리듬으로 움직이고 있는지 확인했습니다. 때로는 폭발적인 커밋 수(예: `SEO` 관련 작업의 142건 커밋)가 필요할 때가 있고, 때로는 정교한 설계와 문서화에 집중해야 할 때가 있습니다. 이 리듬을 잘 조절하는 것이 바로 시니어 엔지니어의 역량이라고 믿습니다.

내일은 또 어떤 데이터가 저의 성장 곡선을 그려낼지 기대됩니다.
