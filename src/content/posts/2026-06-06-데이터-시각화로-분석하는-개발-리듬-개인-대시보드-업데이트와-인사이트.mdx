# [Dev Log] 데이터의 흐름으로 읽는 나의 개발 리듬: 대시보드 업데이트와 인사이트

안녕하세요, 송주한(Jeffrey Joo-Han Song)입니다.

저는 20년 넘게 엔지니어로 일하며 수많은 시스템을 설계하고 구축해왔지만, 여전히 가장 흥미로운 작업 중 하나는 '나의 활동 데이터를 시각화하는 것'입니다. 복잡한 블록체인 아키텍처나 AI 모델을 다루는 일도 즐겁지만, 가끔은 내가 어디에 시간을 쓰고 있는지, 어떤 리듬으로 코드를 작성하고 있는지를 객관적인 수치로 확인하는 과정이 필요하기 때문입니다.

오늘은 개인 대시보드의 데이터를 업데이트하며, 최근의 개발 흐름을 정리하는 시간을 가졌습니다. 단순한 `chore` 작업처럼 보일 수 있지만, 그 이면에는 엔지니어로서 고민하는 '우선순위'와 '디테일'에 대한 이야기가 담겨 있습니다.

### 오늘의 작업: 대시보드 데이터 동기화

이번 커밋의 핵심은 `src/data/dashboard.json` 파일의 업데이트입니다. 자동화된 스크립트를 통해 최근 7일 및 30일간의 커밋 활동과 최신 메시지를 갱신했습니다.

```json
// 변경된 부분 중 일부
{
  "name": "company-page",
  "last_date": "2026-06-05",
  "last_msg": "태블릿(768~1023px) 헤더 깨짐 — nav 브레이크포인트 md→lg 전환",
  "branch": "dev",
  "activity": [0, 0, 0, 0, 10, 20, 0],
  "commits_7d": 30,
  "commits_30d": 30
}
```

### 기술적 결정과 인사이트: "디테일이 완성도를 만든다"

이번 업데이트 내역 중 특히 눈에 띄는 부분은 씨피랩스의 컴퍼니 페이지(`company-page`) 작업입니다. 단순히 기능을 구현하는 것을 넘어, 태블릿 환경에서의 UI 깨짐 현상을 해결하기 위해 브레이크포인트를 `md`에서 `lg`로 전환하는 세밀한 조정 작업을 진행했습니다.

여기서 제가 얻은 인사이트는 **'경계값의 중요성'**입니다. Web3나 DID 같은 고도의 기술적 도메인에서도 마찬가지지만, 결국 사용자가 느끼는 서비스의 퀄리티는 768px과 1023px 사이의 그 미묘한 간극을 어떻게 처리하느냐와 같은 '디테일'에서 결정됩니다. 

또한, `activity` 배열의 수치를 보면 특정 시점에 작업이 집중된 것을 볼 수 있습니다. 엔지니어에게는 몰입(Flow)의 시간이 필요합니다. 며칠간의 탐색과 설계 끝에 단 하루, 이틀 사이에 폭발적으로 커밋이 일어나는 패턴은 제가 문제를 해결하는 전형적인 방식이기도 합니다.

### 마치며

데이터를 업데이트하며 다시 한번 느낀 점은, 기록되지 않은 노력은 기억되지 않는다는 것입니다. 20년의 경력 동안 제가 배운 가장 큰 교훈 중 하나는 **'측정할 수 없으면 개선할 수 없다'**는 것입니다. 

비록 오늘은 JSON 파일 하나를 수정하는 작은 작업이었지만, 이 기록들이 모여 저의 개발 궤적을 만들고, 더 나은 엔지니어링 결정을 내리는 밑거름이 됩니다. 내일은 또 어떤 문제를 풀고, 어떤 기록을 남기게 될지 기대됩니다.
