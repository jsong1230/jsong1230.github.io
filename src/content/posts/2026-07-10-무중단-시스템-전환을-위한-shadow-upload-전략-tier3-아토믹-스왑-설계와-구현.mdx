# [Dev Log] 무중단 시스템 전환을 위한 'Shadow' 전략: Tier3 아토믹 스왑 설계와 구현

안녕하세요, 씨피랩스의 송주한입니다.

엔지니어로서 가장 긴장되는 순간은 '운영 중인 서비스의 핵심 경로(Critical Path)를 변경할 때'일 것입니다. 특히 실시간 스트리밍이나 콘텐츠 서빙처럼 1초의 끊김도 허용되지 않는 환경에서는 더욱 그렇습니다.

오늘은 제가 진행 중인 프로젝트에서 **무중단 서빙 전환(Atomic Swap)**을 위해 도입한 **'Shadow Upload'** 전략과 그 구현 과정에 대해 공유해보려 합니다.

---

### 🛠️ 해결하려던 문제: "어떻게 안전하게 경로를 바꿀 것인가?"

현재 시스템의 HLS(HTTP Live Streaming) 세그먼트 구조를 개선하여 `d<YYYYMMDD>/` 형태의 날짜 기반 경로로 전환해야 하는 과제가 있었습니다. 하지만 단순히 경로를 바꾸고 배포하는 방식은 다음과 같은 리스크가 있습니다.

1. **전환 시점의 공백**: 새 경로로 파일을 올리는 동안 기존 경로를 바라보는 클라이언트는 404 에러를 만납니다.
2. **롤백의 어려움**: 문제가 생겼을 때 즉시 이전 상태로 되돌리려면 물리적인 파일 복구 시간이 필요합니다.

이를 해결하기 위해 저는 **S1(재작성 함수) $\rightarrow$ S2(섀도우 업로드) $\rightarrow$ S3(카나리 플립)**로 이어지는 단계적 전략을 세웠습니다.

### 🚀 오늘의 핵심 작업: S2 Shadow Dated Upload

오늘의 핵심은 **S2 단계인 'Shadow Upload'**의 구현입니다. 핵심 아이디어는 간단합니다. **"실제 서비스가 바라보는 경로는 건드리지 않고, 새 경로에 파일만 미리 다 올려두는 것"**입니다.

#### 1. `sync_shadow_dated.sh` 구현
새로운 경로 구조인 `hls/<era>/d<YYYYMMDD>/`에 세그먼트(`.ts`) 파일들을 additive하게 업로드하는 스크립트를 작성했습니다.

```bash
# sync_shadow_dated.sh 중 일부
# local hls/<MDD>/<era>/cycle_NN_seg_*.ts → S3 hls/<era>/d<YYYYMMDD>/cycle_NN_seg_*.ts
# (additive). manifest 와 live flat 세그먼트는 절대 안 건드림 → prod 서빙 무영향.
```

여기서 중요한 기술적 결정은 **`.m3u8`(매니페스트) 파일은 업로드하지 않는다**는 점입니다. 클라이언트는 매니페스트 파일을 통해 세그먼트 위치를 파악하는데, 매니페스트를 올리지 않으면 S3에 파일이 아무리 많아도 실제 서비스에는 아무런 영향이 없습니다. 즉, **'그림자(Shadow)'**처럼 존재하며 준비만 하는 상태가 되는 것이죠.

#### 2. 검증의 자동화: `verify_prod_sync.py`
단순히 올리는 것보다 중요한 것이 검증입니다. 기존의 flat 구조 검증 로직을 유지하면서, `--dated` 모드를 추가해 새 경로의 파일들이 정확히 업로드되었는지 확인하는 로직을 구현했습니다. 

- **회귀 테스트**: 기존 flat 동작이 불변함을 확인 (5 era $\times$ 6 cycle 검증 완료)
- **실증 테스트**: 특정 era(1985)의 데이터를 업로드하고 `dated verify`가 통과하는지 확인 후, 실제 라이브 서빙에는 영향이 없음을 byte-exact하게 검증했습니다.

### 💡 인사이트: "점진적 전환이 주는 심리적/기술적 안정감"

이번 작업을 통해 다시 한번 느낀 점은, 복잡한 인프라 변경일수록 **'순수 함수'**와 **'단계적 플립(Flip)'**의 조합이 강력하다는 것입니다.

1. **S1(Manifest Rewrite)**: URI를 바꾸는 로직을 순수 함수로 만들어 단위 테스트로 완벽히 검증했습니다.
2. **S2(Shadow Upload)**: 데이터만 미리 배치하여 물리적 전송 리스크를 제거했습니다.
3. **S3(Canary Flip)**: 이제 모든 준비가 끝났으므로, 특정 era(1955) 하나만 매니페스트를 교체(PutObject)하여 카나리 테스트를 진행할 예정입니다.

만약 문제가 발생한다면? S3 Versioning을 통해 이전 매니페스트로 즉시 롤백하면 됩니다. 파일은 이미 양쪽 경로에 다 존재하기 때문에 전환 시간은 밀리초(ms) 단위로 끝납니다.

### 마치며

많은 이들이 '빠른 배포'를 말하지만, 시니어 엔지니어의 미덕은 **'어떻게 하면 가장 안전하게 실패하고, 가장 빠르게 복구할 수 있는 구조를 만드느냐'**에 있다고 생각합니다.
