# [Engineering Log] 200ms의 함정: 비동기 큐와 백그라운드 스로틀링 해결기

안녕하세요, 씨피랩스의 CTO 송주한(Jeffrey Joo-Han Song)입니다.

엔지니어로서 가장 당혹스러운 순간은 "문제를 해결하기 위해 도입한 코드가 더 큰 문제를 일으켰을 때"가 아닐까 합니다. 오늘 제가 겪은 일이 딱 그랬습니다. 모바일 앱의 오디오 트랜지션 과정에서 발생한 레이스 컨디션(Race Condition)을 잡으려다, 예상치 못한 시스템 제약으로 인해 서비스가 '먹통'이 되는 경험을 했습니다.

오늘의 작업 내용을 통해 비동기 처리와 모바일 환경의 특수성에 대해 공유해보고자 합니다.

### 1. 문제의 시작: "시그널이 두 번 나와요"
최근 모바일 앱(v0.9.63)에서 특정 슬롯 경계에서 오디오 시그널이 중복 재생되는 이슈가 있었습니다. 원인을 분석해보니, ExoPlayer가 자연스럽게 다음 트랙으로 넘어가는 시점과 `nonMusicLoader.start`가 호출되어 소스 리스트를 갱신하는 시점이 겹치면서, 이미 재생 중인 트랙을 다시 처음부터 재생하는 'Cold Reset'이 발생했기 때문이었습니다.

이를 해결하기 위해 저는 **'200ms의 대기 시간'**이라는 전략을 세웠습니다.
> "새 리스트를 적용하기 전 200ms만 기다렸다가 다시 확인하자. 그 사이 ExoPlayer가 자연스럽게 다음 트랙에 진입했다면, 굳이 큐를 리셋하지 않고 그대로 진행하면 된다."

### 2. 예상치 못한 부작용: Background JS Throttle
하지만 이 작은 `wait-200ms`가 치명적인 독이 되었습니다. 모바일 환경, 특히 Android의 백그라운드 상태에서는 JS 엔진의 실행 우선순위가 낮아지며 **스로틀링(Throttling)**이 발생합니다.

`enqueueOp`라는 Promise 체인 내부에서 `await`로 대기하던 200ms가, 시스템 스로틀링과 맞물리면서 실제로는 훨씬 더 긴 시간 동안 멈춰버리게 된 것입니다. 결과적으로 `setSrcList` 호출 자체가 락(Lock)이 걸렸고, 07:30분경 '라디오극장' 콘텐츠가 시작되지 않는 사고로 이어졌습니다.

### 3. 기술적 결정: 과감한 Revert와 단계적 접근
사고 보고를 받은 즉시 로그를 분석했고, `slot.transition`은 발생했지만 `setSrcList.enter`가 호출되지 않은 것을 확인했습니다. 해결책은 명확했습니다. 시스템의 불확실성에 기대는 `wait` 전략을 버리는 것이었습니다.

**코드 변경의 핵심:**
```typescript
// v0.9.64 — v0.9.63 의 wait-200ms revert.
// wait-200ms 가 enqueueOp Promise chain 안에서 background JS throttle 로 hang 
// → 자동 transition 시 setSrcList 호출 lock 발생.

// 기존의 복잡한 tryActiveSkip 로직을 제거하고 
// v0.9.62 수준의 Cold Reset 방식으로 회귀 (Revert)
```

결국 v0.9.64에서는 문제가 되었던 대기 로직을 완전히 제거하고, 기존의 안정적인 상태로 되돌렸습니다. 중복 재생 이슈는 `pieces stamp ref`를 통해 부분적으로 커버하고, 나머지 잔존 레이스는 시스템 락을 유발하지 않는 'Phase 2' 방식(액티브 트랙의 위치를 정확히 계산하여 헤드부터 재생)으로 다시 설계하기로 결정했습니다.

### 4. 인사이트: "단순함이 가장 강력한 최적화다"
이번 경험을 통해 다시 한번 깨달은 점은, **비동기 큐(Promise Chain) 내부에서의 임의적인 지연(delay)은 매우 위험하다**는 것입니다. 특히 리소스 제약이 심한 모바일 백그라운드 환경에서는 개발자가 생각하는 '200ms'가 시스템에게는 '무한 대기'가 될 수 있습니다.

기술적인 정교함보다 더 중요한 것은 **예측 가능성(Predictability)**입니다. 엣지 케이스를 잡기 위해 도입한 임시방편이 시스템 전체의 가용성을 해친다면, 그것은 해결책이 아니라 새로운 버그일 뿐입니다.

내일은 더 견고한 'Phase 2' 픽스를 적용해, 안정성과 정확성이라는 두 마리 토끼를 모두 잡도록 하겠습니다.
