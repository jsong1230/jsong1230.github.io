안녕하세요, 송주한(Jeffrey Joo-Han Song)입니다.

엔지니어로서 커리어를 쌓아오며 제가 가장 중요하게 생각하는 것 중 하나는 '가시성(Visibility)'입니다. 복잡한 시스템일수록 현재 어떤 상태인지, 어디서 병목이 발생하는지, 그리고 우리가 계획한 방향으로 나아가고 있는지를 데이터로 확인하는 과정이 필수적이기 때문입니다.

오늘은 제 개인 대시보드의 데이터를 업데이트하는 간단한 작업을 진행했습니다. 겉으로 보기에는 단순한 `JSON` 파일의 수정처럼 보이지만, 그 이면에 담긴 엔지니어링적 관점과 인사이트를 공유해보고자 합니다.

### 데이터 기반의 자기 객관화: 대시보드 업데이트

오늘 작업의 핵심은 `src/data/dashboard.json` 파일의 갱신이었습니다. 저는 제가 관리하는 여러 프로젝트의 커밋 빈도, 최근 활동, 그리고 메시지들을 추적하는 개인적인 대시보드를 운영하고 있습니다.

```json
-  "generated_at": "2026-06-26T23:00:01.561Z",
+  "generated_at": "2026-06-27T23:00:01.817Z",
```

단순히 날짜가 바뀌는 것이 아니라, 지난 일주일간의 활동량(`commits_7d`)과 일별 활동 지표(`activity`)를 최신화하는 과정입니다. 예를 들어, `jsong1230.github.io` 프로젝트의 경우 `commits_30d`가 68에서 69로 증가하며 꾸준한 기록의 힘을 보여주고 있습니다.

### 왜 이런 '사소한' 기록이 중요한가?

CTO로서 조직을 리딩할 때 제가 팀원들에게 강조하는 것 중 하나가 바로 '측정 가능한 성장'입니다. 코드 한 줄을 더 짜는 것보다 중요한 것은, 내가 지금 어떤 리듬으로 일하고 있는지를 파악하는 것입니다.

이번 diff에서 흥미로운 점은 각 프로젝트의 `activity` 배열의 변화입니다. 특정 날짜에 몰려있던 활동이 다른 날로 이동하거나, `commits_7d` 수치가 변동하는 것을 보며 제가 최근 어떤 도메인에 더 많은 에너지를 쏟았는지 직관적으로 알 수 있습니다. 씨피랩스에서의 기술적 도전들과 개인적인 연구, 그리고 오픈소스 기여 사이의 균형을 맞추는 데 이 데이터가 훌륭한 나침반 역할을 합니다.

### 기술적 결정과 인사이트: 정적 데이터의 효율성

물론 이런 데이터를 실시간 API로 처리할 수도 있습니다. 하지만 저는 의도적으로 정적 `JSON` 파일을 업데이트하고 이를 배포하는 방식을 택했습니다.

1. **성능과 비용**: 단순 조회용 대시보드에 무거운 DB 쿼리를 날릴 필요가 없습니다. 정적 파일은 CDN을 통해 전 세계 어디서든 가장 빠르게 서빙됩니다.
2. **불변성(Immutability)**: 특정 시점의 스냅샷이 파일로 남기 때문에, 시간이 흐른 뒤 "내가 2026년 6월에는 어떤 고민을 하며 코딩했는가"를 Git 히스토리를 통해 추적할 수 있습니다.

### 마치며

개발자에게 Git 커밋 로그는 단순한 작업 기록이 아니라, 그 사람의 사고 과정이 담긴 '족적'과 같습니다. 오늘 `chore: Update dashboard data`라는 짧은 커밋 메시지 뒤에는, 스스로의 생산성을 측정하고 더 나은 엔지니어가 되기 위한 루틴이 숨어 있습니다.

여러분도 자신만의 '성장 지표'를 만들어 기록해 보시는 건 어떨까요? 아주 작은 데이터라도 그것이 쌓이면 강력한 인사이트가 됩니다.

오늘도 기록하며 성장하는 하루 되시길 바랍니다.
