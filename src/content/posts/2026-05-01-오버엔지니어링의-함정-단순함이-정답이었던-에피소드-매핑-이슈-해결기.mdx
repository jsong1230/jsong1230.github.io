# [Dev Log] 오버엔지니어링의 함정과 '단순함'의 미학: 에피소드 매핑 이슈 해결기

안녕하세요, 씨피랩스의 CTO 송주한입니다.

개발을 하다 보면 "미래의 편의성"이나 "특수한 상황"을 대비해 미리 장치를 만들어두곤 합니다. 하지만 때로는 그 '배려'가 시스템의 복잡도를 높이고, 예상치 못한 버그의 근원이 되기도 하죠. 오늘 제가 겪은 에피소드 매핑 이슈가 딱 그랬습니다.

### 1. 문제의 발단: 모든 날짜가 '오늘'로 수렴하는 현상

최근 진행 중인 프로젝트에서 모바일 다이얼을 통해 특정 날짜의 콘텐츠를 요청하는 기능을 구현했습니다. 그런데 테스트 과정에서 이상한 점이 발견되었습니다. 5월 11일부터 15일까지 어떤 날짜를 선택해도, 서버는 계속해서 동일한(오늘 날짜에 매핑된) 오디오 파일만 반환하는 'mismatch' 현상이 발생한 것입니다.

데이터베이스나 mp3 파일 자체에는 문제가 없었습니다. 원인은 서버 사이드에 심어두었던 **`LAUNCH_PREVIEW_MAP`**이라는 강제 매핑 로직에 있었습니다.

### 2. 기술적 결정: "더하는 것보다 빼는 것이 정답일 때"

처음 이 맵을 도입한 의도는 좋았습니다. 정식 런칭(D-Day) 전이라도 사용자가 접속한 요일에 맞춰 매일 다른 콘텐츠를 미리 보여주고 싶었기 때문입니다.

```python
# Problematic Code: 서버에서 클라이언트의 요청 날짜를 무시하고 강제로 덮어씀
LAUNCH_PREVIEW_MAP: dict[int, tuple[int, int]] | None = {
    0: (5, 11),  # 월요일이면 무조건 5/11 콘텐츠
    1: (5, 12),  # 화요일이면 무조건 5/12 콘텐츠
    ...
}
```

하지만 클라이언트가 정확한 캘린더 날짜(`year, month, day`)를 보내기 시작하면서, 서버의 강제 매핑 로직이 클라이언트의 요청 값을 덮어써 버리는 충돌이 발생했습니다. 특히 서버 컨테이너의 타임존(UTC) 설정까지 맞물리면서, 사용자가 요청한 날짜와 상관없이 서버가 계산한 '오늘의 요일'에 해당하는 단 하나의 콘텐츠로 수렴해버린 것이죠.

여기서 제가 내린 결정은 간단했습니다. **이 로직을 완전히 제거(None 처리)하는 것**이었습니다.

```python
# Fix: 강제 매핑을 폐기하고 클라이언트의 요청 값을 그대로 신뢰함
LAUNCH_PREVIEW_MAP: dict[int, tuple[int, int]] | None = None
```

### 3. 이번 작업에서 배운 인사이트

이번 수정으로 `resolve_md`나 `resolve_weekday` 같은 하위 함수들의 시그니처를 변경하지 않고도 문제를 근본적으로 해결할 수 있었습니다. 호출부의 코드를 수정할 필요 없이 설정값 하나로 동작을 제어했기에 리스크도 최소화했습니다.

이번 경험을 통해 다시 한번 느낀 점은 **"서버는 가능한 한 Stateless하고 투명해야 한다"**는 것입니다. 클라이언트가 명시적으로 특정 날짜를 요청했다면, 서버가 임의의 비즈니스 로직으로 이를 가공하기보다 요청된 값을 그대로 처리하는 것이 가장 예측 가능하고 견고한 설계입니다.

만약 '미리보기' 기능이 다시 필요하다면, 그것은 서버의 강제 매핑이 아니라 클라이언트가 `today`를 `preview-day`로 변환해서 요청하는 방식으로 풀어내는 것이 맞습니다.

### 마치며

엔지니어로서 우리는 항상 더 스마트한 기능을 추가하고 싶어 합니다. 하지만 때로는 **불필요한 로직을 걷어내는 '삭제의 용기'**가 시스템의 안정성을 높이는 가장 빠른 길이라는 것을 다시금 깨달은 하루였습니다.

오늘의 트러블슈팅은 여기서 마칩니다. 내일은 더 단순하고 강력한 코드로 찾아뵙겠습니다.
