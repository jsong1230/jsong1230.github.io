# [Dev Log] 픽셀 하나가 만드는 사용자 경험: iPhone 17 대응과 Safe Area의 이해

안녕하세요, 씨피랩스(CP Labs)의 CTO 송주한입니다.

엔지니어로 오래 일하다 보면 거대한 아키텍처 설계나 복잡한 비즈니스 로직을 푸는 일만큼이나, 때로는 '단 몇 픽셀'의 오차를 잡는 일이 얼마나 중요한지 깨닫게 됩니다. 특히 다양한 디바이스 환경을 지원해야 하는 프론트엔드 작업에서는 더욱 그렇습니다.

오늘은 제가 진행한 작은 수정 사항이지만, 모바일 웹 환경에서 흔히 겪는 **'Safe Area'** 문제와 그 해결 과정에 대해 공유해보려 합니다.

### 📱 문제 상황: 사라진 '편성표 보기' 버튼

최근 iPhone 17 Pro 시뮬레이터(iOS 26.4)에서 서비스 화면을 테스트하던 중, 하단의 '편성표 보기' 토글 버튼이 완전히 가려져 보이지 않는 현상을 발견했습니다. 

원인을 분석해보니, 기존에 하드코딩되어 있던 `.shell` 클래스의 `padding-bottom: 110px`가 문제였습니다. 최신 기기일수록 하단 홈 바(Home Indicator) 영역을 보호하기 위한 **safe-area-inset-bottom** 값이 커지는데, 기존 설정은 이 동적인 영역과 볼륨 컨트롤러(`.vol`)의 실제 높이를 충분히 수용하지 못했던 것입니다. 계산 결과 약 38px 정도가 부족해 버튼이 그라데이션 하단으로 깔려버린 상태였습니다.

### 🛠️ 기술적 결정: 하드코딩에서 동적 계산으로

단순히 `padding-bottom` 값을 150px로 높이는 방법도 있었겠지만, 이는 구형 기기나 일반 웹 브라우저에서 불필요하게 넓은 공백을 만드는 결과를 초래합니다. 저는 **`calc()`** 함수와 **`env()`** 변수를 조합해 기기별 최적값을 갖도록 수정했습니다.

```css
/* 수정 전 */
padding-bottom: 110px;

/* 수정 후 */
padding-bottom: calc(140px + env(safe-area-inset-bottom, 0px));
```

**여기서 핵심은 두 가지입니다.**
1. **`140px`**: 볼륨 컨트롤러의 구성 요소(패딩, 라벨, 슬라이더 등)의 전체 높이 상한선에 10px의 여유분을 더한 값입니다.
2. **`env(safe-area-inset-bottom, 0px)`**: iOS의 Safe Area 값을 동적으로 가져옵니다. Safe Area가 없는 환경에서는 `0px`가 적용되어 기존 대비 약 30px 정도의 적절한 패딩만 추가되므로, 이전 세대 기기나 데스크톱 Safari에서도 레이아웃 깨짐 없이 안정적으로 작동합니다.

### 💡 인사이트: '디테일'이 곧 '품질'이다

이번 작업은 코드 한 줄의 변경이었지만, 사용자 입장에서는 **'기능이 작동하지 않는 서비스'**와 **'완벽하게 작동하는 서비스'**를 가르는 결정적인 차이였습니다. 

특히 Web3나 AI 서비스처럼 최첨단 기술을 다루는 제품일수록, 사용자가 처음 마주하는 UI의 완성도는 제품 전체의 신뢰도로 이어집니다. 씨피랩스에서 추구하는 엔지니어링의 가치 또한 이러한 디테일한 완성도에서 시작된다고 믿습니다.

최신 디바이스가 출시될 때마다 레이아웃을 점검하는 것은 번거로운 일일 수 있습니다. 하지만 `env()`와 같은 표준 CSS 변수를 적극 활용해 유연한 대응 체계를 구축해둔다면, 유지보수 비용을 획기적으로 줄일 수 있을 것입니다.

오늘도 픽셀 하나, 코드 한 줄에 진심을 담아 개발을 이어갑니다.
