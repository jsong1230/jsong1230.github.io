# [Dev Log] 상태 동기화의 함정과 디테일의 차이: HLS 빌더 최적화 작업

안녕하세요, 씨피랩스의 CTO 송주한입니다.

개발자로서 가장 긴장되는 순간 중 하나는 '잘 돌아가고 있다고 믿었던' 시스템에서 예상치 못한 엣지 케이스가 발견되었을 때입니다. 특히 분산 환경이나 클라우드 인프라 위에서 동작하는 서비스라면, 로컬에서는 절대 재현되지 않는 '상태(State)의 불일치' 문제가 발생하곤 하죠.

오늘은 제가 진행한 HLS(HTTP Live Streaming) 빌더의 핫픽스와 오디오 믹싱 디테일 조정 작업을 통해 느낀 점들을 공유해 보려 합니다.

### 1. Ephemeral Storage의 함정과 상태 동기화 (S3 Marker Sync)

오늘 가장 핵심적인 작업은 S3 동기화 로직의 수정이었습니다. 기존 로직은 `.m3u8`과 `.ts` 파일만 S3에 업로드하고, 빌드 날짜를 기록하는 `.build_date` 마커 파일은 명시적으로 스킵하고 있었습니다.

하지만 여기서 문제가 발생했습니다. ECS(Elastic Container Service) 태스크가 재기동되면 컨테이너 내부의 ephemeral storage(임시 저장소)에 있던 마커 파일이 사라집니다. 이때 시스템은 S3에 저장된 이전 마커를 참조하게 되는데, 만약 S3의 마커가 오래되었다면(stale), 빌드 사이클 스킵 로직이 오작동하여 불필요한 재빌드가 일어나거나 동기화가 꼬이는 현상이 재현된 것입니다.

**코드 변경 포인트:**
```python
# 기존: .m3u8 + .ts 만 업로드 (marker 등은 skip)
# 변경: .build_date를 추가하여 S3가 Audit Source가 되도록 수정
if p.name == ".build_date":
    content_type = "text/plain"
else:
    ext = p.suffix.lower()
    if ext not in _CONTENT_TYPE_BY_EXT:
        continue
    content_type = _CONTENT_TYPE_BY_EXT[ext]
```

단순히 파일 하나를 더 올리는 변경이었지만, 이를 통해 **"인프라의 휘발성(Ephemerality)을 인정하고, 신뢰할 수 있는 단일 상태 저장소(Single Source of Truth)를 S3로 일원화"**하는 결정을 내렸습니다. 인프라 레벨의 특성을 고려하지 않은 소프트웨어 설계가 어떻게 런타임 오류로 이어지는지를 다시 한번 배운 사례였습니다.

### 2. 0.1초의 디테일: Acrossfade 통일

기술적인 핫픽스 외에, 사용자 경험(UX)을 결정짓는 오디오 믹싱 작업도 진행했습니다. 서로 다른 오디오 소스(Voice, Music, Tail)가 전환될 때 겹치는 구간인 `acrossfade` 시간을 조정하는 작업이었습니다.

청취 피드백을 통해 `voice → tail` 구간의 1.5초 전환이 다소 거칠게 들린다는 점을 확인했고, 이를 3.0초로 상향했습니다. 더 나아가 음악 슬롯 체인 내의 모든 전환(`tail → music`, `music → tail` 등)을 3.0초로 통일하여 일관된 부드러움을 확보했습니다.

```python
_XFADE_DUR_BY_PAIR = {
    ("voice", "tail"): 3.0,
    ("voice", "music"): 3.0,
    ("tail", "tail"): 3.0,   # 2.5s -> 3.0s
    ("tail", "music"): 3.0,  # 2.0s -> 3.0s
    ("music", "music"): 3.0,
    ("music", "tail"): 3.0,  # 2.0s -> 3.0s
}
```

엔지니어링은 단순히 기능을 구현하는 것이 아니라, 도메인 전문가(이번 경우 김 이사님)의 피드백을 정교한 파라미터로 변환하여 시스템에 녹여내는 과정임을 다시금 느꼈습니다.

### 마치며: '정석'대로 하는 것의 가치

오늘 작업 중 흥미로웠던 점은 배포 방식에 대한 고민이었습니다. 급한 수정이라 `docker cp`로 빠르게 적용할 수도 있었지만, 이는 이미지 베이크(baked) 시 이전 값으로 회귀할 위험이 큽니다. 결국 `main` 브랜치 커밋 후 프로덕션 이미지를 다시 빌드하는 '정석'적인 경로를 택했습니다.

속도보다 중요한 것은 **추적 가능성(Traceability)**과 **재현 가능성(Reproducibility)**입니다. 20년 넘게 엔지니어로 일하며 내린 결론은, 결국 기본을 지키는 것이 가장 빠른 길이라는 점입니다.

오늘도 한 뼘 더 성장한 시스템과 함께 하루를 마무리합니다.
