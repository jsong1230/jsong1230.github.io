# [Dev Log] 데이터의 흐름으로 기록하는 나의 성장 궤적

안녕하세요, 씨피랩스(CP Labs)의 CTO 송주한(Jeffrey)입니다.

엔지니어로서 커리어를 쌓아오며 제가 가장 중요하게 생각하는 것 중 하나는 '기록'입니다. 단순히 코드를 짜는 것을 넘어, 내가 어떤 고민을 했고 어떤 리듬으로 움직였는지를 시각화하는 것은 개인의 성장은 물론 팀의 생산성을 관리하는 데 매우 중요한 인사이트를 제공하기 때문입니다.

오늘은 제 개인 대시보드(`jsong1230.github.io`)의 데이터를 업데이트하며, 최근의 작업 흐름을 되짚어보는 시간을 가졌습니다.

### 단순한 업데이트, 그 이상의 의미

오늘의 작업은 겉보기에 `dashboard.json` 파일의 데이터를 갱신하는 단순한 `chore` 작업처럼 보일 수 있습니다. 하지만 이 JSON 데이터에는 제가 최근 집중하고 있는 프로젝트들의 '심박수'가 담겨 있습니다.

이번 diff에서 특히 흥미로운 부분은 `heritageradio` 레포지토리의 활동성 변화입니다.

```json
{
  "name": "heritageradio",
  "last_date": "2026-07-16",
  "last_msg": "fix(admin): 품질감사 Beat prod 게이트 (ENV != prod 일 때만 등록)",
  "branch": "dev",
  "activity": [0, 0, 3, 12, 0, 26, 0],
  "commits_7d": 58,
  "commits_30d": 101
}
```

최근 7일간 58개의 커밋이 발생했고, 30일 기준으로는 101개의 커밋이 쌓였습니다. 특히 `last_msg`에 기록된 **"품질감사 Beat prod 게이트"** 관련 수정 사항은 실무에서 매우 중요한 기술적 결정이 포함된 부분입니다. 운영 환경(`prod`)과 개발 환경의 격리를 엄격히 하여, 예기치 못한 데이터 오염이나 설정 오류가 실제 서비스에 영향을 주지 않도록 가드레일을 세우는 작업이었죠.

### 기술적 결정: 정적 데이터의 동적 관리

제 대시보드는 복잡한 백엔드 서버 없이 JSON 파일 하나로 구동되는 정적 구조를 가지고 있습니다. 누군가는 "왜 굳이 수동(혹은 스크립트)으로 JSON을 업데이트하느냐"고 물을 수 있습니다.

하지만 저는 여기서 **'단순함의 미학'**을 찾습니다. 
1. **오버헤드 최소화**: 대시보드를 위해 별도의 DB와 API 서버를 운영하는 것은 배보다 배꼽이 더 큰 격입니다.
2. **버전 관리**: 데이터 자체가 Git 커밋으로 남기 때문에, 특정 시점의 내 활동량을 `git checkout` 한 번으로 되돌려 볼 수 있습니다.
3. **불변성**: 한 번 기록된 데이터는 쉽게 변하지 않으며, 이는 곧 신뢰할 수 있는 아카이브가 됩니다.

### 마치며: 엔지니어의 리듬

`claude-agent`나 `cplabs_homepage` 같은 다양한 프로젝트들이 동시에 돌아가고 있지만, 각 프로젝트의 `activity` 배열을 보면 저만의 작업 리듬이 보입니다. 어떤 날은 폭발적으로 코드를 작성하고, 어떤 날은 전략적인 미팅과 설계에 집중하며 커밋 수가 줄어듭니다.

중요한 것은 '매일 얼마나 많은 코드를 썼느냐'가 아니라, **'지금 내가 풀고 있는 문제가 무엇이며, 그것이 올바른 방향으로 해결되고 있는가'**입니다.

오늘의 데이터 업데이트를 통해 다시 한번 제 리듬을 점검했습니다. 내일은 또 어떤 문제들을 풀며 이 JSON 파일에 의미 있는 기록을 남기게 될지 기대됩니다.

**Jeffrey's Insight:** 
*최고의 도구는 가장 복잡한 도구가 아니라, 내 목적에 가장 부합하며 유지보수 비용이 가장 적은 도구입니다.*
