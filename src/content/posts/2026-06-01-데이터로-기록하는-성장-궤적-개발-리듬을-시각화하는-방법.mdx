# [Dev Log] 데이터의 흐름으로 기록하는 나의 개발 리듬

안녕하세요, 송주한(Jeffrey)입니다.

개발자에게 있어 '기록'은 단순히 과거를 남기는 것이 아니라, 자신의 성장 궤적을 확인하고 현재의 리듬을 점검하는 중요한 과정입니다. 저는 개인적인 개발 활동과 생산성을 시각화하기 위해 대시보드를 운영하고 있는데, 오늘은 그 기반이 되는 `dashboard.json` 데이터를 업데이트했습니다.

단순한 데이터 갱신처럼 보일 수 있는 `chore: Update dashboard data` 커밋이지만, 그 이면에는 제가 데이터를 통해 나 자신을 어떻게 모니터링하고 있는지에 대한 고민이 담겨 있습니다.

### 데이터로 보는 개발 리듬: 'Activity'의 의미

이번 업데이트에서 가장 흥미로운 부분은 각 리포지토리별 `activity` 배열의 변화입니다. 

```json
"activity": [
  2,
  2,
  2,
  2,
  3,
  3,
  0
]
```

이 숫자들이 의미하는 것은 단순한 커밋 횟수가 아니라, 일주일 단위로 쪼개어 본 나의 '몰입도'입니다. 특정 프로젝트에 치우치지 않고 여러 리포지토리를 오가며 컨텍스트 스위칭을 하고 있는지, 혹은 하나의 핵심 과제에 깊게 몰입하고 있는지를 한눈에 파악할 수 있죠.

이번 데이터를 보면 `heritageradio` 프로젝트의 커밋 수(`commits_30d: 427`)가 압도적으로 높습니다. 이는 최근 씨피랩스에서 진행 중인 핵심 모듈 추출 작업과 엔드포인트 최적화 등 기술적 난도가 높은 작업에 집중했음을 보여줍니다. 특히 `aired_guard`를 공용 모듈로 추출하는 과정에서 겪은 설계적 고민들이 이 수치에 고스란히 녹아있습니다.

### 기술적 결정: 자동화와 정적 데이터의 조화

저는 이 대시보드를 위해 복잡한 DB를 구축하는 대신, JSON 파일 기반의 정적 데이터 구조를 선택했습니다. 이유는 명확합니다.

1. **오버헤드 최소화**: 개인 대시보드에 과도한 인프라를 투입하는 것은 낭비입니다.
2. **버전 관리**: 데이터의 변화 자체가 Git 히스토리에 남기 때문에, 언제 내 생산성이 변했는지 `diff`를 통해 직관적으로 확인할 수 있습니다.
3. **배포 효율**: GitHub Pages와 같은 정적 호스팅 환경에서 가장 빠르게 렌더링될 수 있습니다.

### 인사이트: 측정할 수 없으면 개선할 수 없다

피터 드러커의 말처럼, 측정할 수 없는 것은 개선할 수 없습니다. 20년 넘게 엔지니어로 일하며 깨달은 점은, 감각에 의존한 '열심히 했다'는 느낌보다 객관적인 '데이터'가 주는 피드백이 훨씬 강력하다는 것입니다.

오늘의 `dashboard.json` 업데이트는 단순한 숫자 변경이 아니라, 지난 일주일간 씨피랩스에서 어떤 기술적 도전을 했고, 개인 프로젝트에서는 어떤 리듬을 유지했는지 복기하는 시간이었습니다. 

여러분은 자신의 성장과 리듬을 어떻게 기록하고 계신가요? 거창한 도구가 아니더라도, 나만의 지표를 만들어 기록해 보시길 추천합니다. 그 작은 데이터들이 모여 결국 대체 불가능한 엔지니어의 커리어를 만든다고 믿습니다.
