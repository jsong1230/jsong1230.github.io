# [Dev Log] 사용자 경험(UX)의 디테일: 자동 선택과 캐시 버스팅

안녕하세요, 씨피랩스의 CTO 송주한(Jeffrey)입니다.

엔지니어로서 복잡한 아키텍처를 설계하는 것도 중요하지만, 때로는 아주 작은 코드 변경이 사용자가 느끼는 서비스의 완성도를 결정짓곤 합니다. 오늘은 관리자 페이지의 운영 효율을 높이기 위해 진행한 두 가지 작은, 하지만 임팩트 있는 개선 작업에 대해 공유해보려 합니다.

### 1. "매번 클릭하시나요?" — KST 기준 주차 자동 선택

관리자 페이지에서 에피소드를 등록하는 `episode-registry` 기능이 있습니다. 기존에는 페이지에 진입하면 항상 첫 번째 탭(5월 1주차)이 기본으로 선택되어 있었습니다. 하지만 운영자 입장에서는 오늘 날짜에 해당하는 주차의 데이터를 수정하는 일이 대부분입니다. 매번 탭을 찾아 클릭하는 행위는 사소해 보이지만, 반복되면 상당한 피로감을 줍니다.

이를 해결하기 위해 **KST(한국 표준시) 기준 오늘 날짜가 포함된 주차를 자동으로 계산해 초기값으로 설정**하는 로직을 추가했습니다.

```typescript
function findCurrentWeekIndices(): { monthIdx: number; weekInMonthIdx: number } {
  const nowUtc = new Date();
  const kstOffsetMs = 9 * 60 * 60 * 1000;
  const nowKst = new Date(nowUtc.getTime() + kstOffsetMs);
  const todayMonth = nowKst.getUTCMonth() + 1;
  const todayDay = nowKst.getUTCDate();
  
  for (let mi = 0; mi < MONTHS.length; mi++) {
    if (MONTHS[mi].month !== todayMonth) continue;
    for (let wi = 0; wi < MONTHS[mi].weeks.length; wi++) {
      if (MONTHS[mi].weeks[wi].days.some(d => d.month === todayMonth && d.day === todayDay)) {
        return { monthIdx: mi, weekInMonthIdx: wi };
      }
    }
  }
  return { monthIdx: 0, weekInMonthIdx: 0 }; // Fallback
}
```

단순히 `new Date()`를 쓰지 않고 KST 오프셋을 명시적으로 처리한 이유는 서버 환경이나 브라우저 설정에 관계없이 한국 시간 기준으로 정확한 주차를 매칭하기 위함입니다. `useMemo`를 통해 마운트 시점에 한 번만 계산하도록 최적화하여, 불필요한 리렌더링 없이 매끄러운 진입 경험을 구현했습니다.

### 2. "왜 업데이트가 안 보이지?" — iframe 캐시 버스팅(Cache-Busting)

또 다른 문제는 `LivePlayerPanel`에서 발생했습니다. HLS 플레이어(`player.html`)는 주차 업데이트나 UI 변경이 잦은 페이지인데, 브라우저가 이 HTML 파일을 강하게 캐싱하면서 최신 변경 사항이 반영되지 않는 사고가 있었습니다.

가장 빠르고 확실한 해결책은 URL 뒤에 매번 다른 쿼리 스트링을 붙여 브라우저가 이를 '새로운 요청'으로 인식하게 만드는 **캐시 버스팅** 기법입니다.

```typescript
function LivePlayerPanel() {
  // tab mount 시점 timestamp 를 query 로 붙여 브라우저 cache 우회
  const [bust] = useState<string>(() => `?_=${Date.now()}`);
  const src = `${LIVE_PLAYER_URL}${bust}`;
  
  return (
    <iframe src={src} ... />
  );
}
```

`useState`의 lazy initialization(함수 형태의 초기값)을 사용하여, 컴포넌트가 마운트되는 시점의 타임스탬프를 딱 한 번만 생성하도록 했습니다. 덕분에 운영자는 새로고침 없이도 항상 최신 버전의 플레이어를 확인할 수 있게 되었습니다.

### 마치며: 엔지니어의 시선

오늘의 작업은 거창한 알고리즘이나 새로운 프레임워크 도입은 아니었습니다. 하지만 **"사용자가 어디서 불편함을 느끼는가"**와 **"브라우저의 기본 동작(캐싱)이 비즈니스 로직에 어떤 영향을 주는가"**를 고민한 결과물입니다.

씨피랩스에서 추구하는 기술적 지향점 역시 이와 같습니다. 깊이 있는 기술 스택을 다루는 것만큼이나, 실제 제품을 사용하는 사람이 느끼는 '매끄러움'을 구현하는 것이 진정한 엔지니어링의 완성이라고 믿습니다.

작은 디테일이 모여 위대한 제품이 됩니다. 오늘도 즐거운 코딩 하세요!
