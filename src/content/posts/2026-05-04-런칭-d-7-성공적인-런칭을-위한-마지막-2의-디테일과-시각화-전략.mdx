# [Dev Log] 런칭 D-7, 디테일의 완성도가 성패를 가른다

안녕하세요, 씨피랩스의 송주한입니다.

프로젝트 런칭이 일주일 앞으로 다가왔습니다. 개발자로서 가장 긴장되면서도 몰입도가 높아지는 시기죠. 기능 구현이 거의 완료된 이 시점에는 거창한 아키텍처 설계보다, 사용자가 실제로 마주하게 될 '엣지 케이스'를 잡고 운영 효율성을 높이는 '디테일'에 집중해야 합니다.

오늘 제가 진행한 작업들은 바로 그 '마지막 2%'를 채우는 과정이었습니다.

### 1. 커뮤니케이션의 시각화: D-7 런칭 체크리스트
런칭 직전에는 수많은 유관 부서(인프라, 콘텐츠, 마케팅, 법무 등)가 동시에 움직입니다. 이때 가장 위험한 것이 '서로 다르게 이해하고 있는 것'입니다. 텍스트 기반의 문서도 좋지만, 저는 이번 회의를 위해 **HTML 기반의 시각화 브리핑 자료**를 직접 작성했습니다.

단순한 리스트가 아니라 캘린더 형태의 타임라인과 책임자별 분담 표를 구현하여, 누가 언제까지 무엇을 해야 하는지 한눈에 들어오게 만들었습니다.

```html
<!-- 런칭 카운트다운과 타임라인을 강조한 스타일링 -->
.header .countdown {
  display: inline-block;
  margin-top: 16px;
  padding: 6px 18px;
  background: var(--accent);
  color: #fff;
  border-radius: 999px;
  font-size: 13px;
  font-weight: 700;
}
```

기술적으로는 간단한 HTML/CSS 작업이지만, CTO로서 제가 생각하는 '기술'은 단순히 코드를 짜는 것이 아니라 **문제를 해결하는 최적의 도구를 선택하는 것**입니다. 복잡한 협업 툴보다 때로는 잘 짜인 단일 HTML 페이지 하나가 의사결정 속도를 획기적으로 높여줍니다.

### 2. 운영의 안정성: 엣지 케이스 대응과 데이터 복구
오늘 작업 중 가장 중요했던 부분은 실제 운영 환경에서 발생한 '사고'들에 대한 수정이었습니다.

*   **자정 자동 리프레시 이슈:** 모바일 앱에서 자정에 데이터가 제대로 갱신되지 않는 버그를 수정했습니다. 서비스 성격상 시간 기반의 데이터 업데이트가 핵심인데, 이 부분의 누락은 치명적일 수 있습니다.
*   **콘텐츠 복구:** `JOSEON_STORY`, `YASA_STORY` 등 마스터 데이터의 일부 누락분(Batch 2)을 회복시켰습니다. 대량의 콘텐츠를 다루다 보면 예상치 못한 데이터 유실이 발생하곤 하는데, 이를 빠르게 트래킹하고 복구하는 파이프라인의 중요성을 다시금 느꼈습니다.
*   **인프라 설정:** `LAUNCH_PREVIEW_MAP` 활성화 및 특정 음악 재생 사고(BODY_EXERCISE)를 수정하며 프로덕션 환경의 안정성을 확보했습니다.

### 3. 오늘의 인사이트: "완성도는 수정의 반복에서 온다"
오늘의 커밋 로그를 보면 버전 수정(`v0.9.40`), 누락된 파일 픽스, 메뉴 숨김 처리 등 소소한 변경사항이 많습니다. 누군가는 이를 '단순 작업'이라고 생각할 수 있겠지만, 20년 넘게 엔지니어링을 하며 배운 점은 **결국 사용자가 느끼는 퀄리티는 이런 작은 디테일들의 합**이라는 것입니다.

특히 런칭 직전의 `package.json`이나 `version.ts` 같은 설정 파일 하나가 배포 전체를 꼬이게 만들 수 있습니다. "이 정도면 됐겠지"라는 생각보다 "여기서 무엇이 더 잘못될 수 있을까"를 고민하는 태도가 안정적인 런칭을 만듭니다.

이제 정말 7일 남았습니다. 씨피랩스의 팀원들과 함께 마지막까지 꼼꼼하게 점검하여 성공적인 런칭을 만들어내겠습니다.
