# [Dev Log] '알람 피로'를 줄이는 디테일: 날짜 경계 조건(Date-Race) 해결하기

안녕하세요, 씨피랩스의 CTO 송주한입니다.

엔지니어로서 가장 경계해야 할 것 중 하나가 바로 **'알람 피로(Alarm Fatigue)'**입니다. 시스템에서 계속해서 경고가 발생하지만, 정작 실무자가 "아, 이건 원래 저 시간에 나는 가짜 알람이야"라고 치부하게 되는 순간, 진짜 심각한 장애가 발생해도 그것이 노이즈에 묻혀 발견되지 못하는 치명적인 리스크가 발생하기 때문입니다.

오늘은 제가 운영 중인 자동화 스크립트에서 발생하던 전형적인 '날짜 경계 조건' 문제와 이를 해결한 과정을 공유하려 합니다.

### 문제의 발단: 22시의 미스터리

최근 특정 시간대(22시~24시)에만 매시간 검증 실패 알람이 발생하는 현상이 있었습니다. 원인을 분석해 보니, 콘텐츠 동기화 로직과 검증 로직 사이의 **'시점 차이'**가 문제였습니다.

시스템은 매일 22:00에 내일(Tomorrow)의 콘텐츠를 미리 S3에 올리는 `sync_tomorrow` 작업을 수행합니다. 하지만 시간별로 상태를 체크하는 `verify_prod_sync_hourly.sh` 스크립트는 기본적으로 '오늘(Today)'의 날짜를 기준으로 검증을 수행하고 있었습니다.

결과적으로 **22시 이후에는 S3에 이미 '내일'의 데이터가 올라와 있는데, 검증기는 '오늘'의 데이터가 맞는지 확인하니 당연히 불일치(Mismatch)가 발생**하고, 이것이 매시간 알람으로 이어진 것입니다.

### 기술적 결정: 검증 그룹의 분리 (Split Verification)

단순히 해당 시간대의 알람을 끄는 것은 위험합니다. 진짜 데이터 누락이 발생했을 때 알 수 없기 때문입니다. 저는 검증 로직을 **'오늘 그룹'**과 **'내일 그룹'**으로 분리하여 정밀하게 검증하는 방식을 택했습니다.

**핵심 변경 사항:**
1. **윈도우 설정:** 22시~23시 사이에는 검증 대상을 두 그룹으로 나눕니다.
2. **Advanced Cycle 처리:** 내일 날짜로 미리 업데이트된 사이클(Advanced Cycle)은 `TOMORROW` 날짜 기준으로 검증합니다.
3. **Tail Cycle 처리:** 아직 업데이트되지 않은 나머지 사이클은 기존처럼 `TODAY` 기준으로 검증합니다.

코드에서 구현한 로직의 핵심은 다음과 같습니다.

```bash
# advanced cycle = sync_tomorrow 가 22:00 에 올린 cycle 집합 (tomorrow 요일 기준)
if (( HOUR >= 22 )); then
  # 내일이 주말인지 평일인지에 따라 advance 대상 cycle 결정
  if [[ "$TOMORROW_WD" = "6" || "$TOMORROW_WD" = "7" ]]; then
    ADVANCED=(00)
  else
    ADVANCED=(00 01 02 03 04)
  fi
  TOMORROW_CYCLES=("${ADVANCED[@]}")
  # TODAY_CYCLES = TODAY_FULL - ADVANCED (차집합 처리)
  ...
fi
```

이렇게 분리한 후 `run_group`이라는 함수를 통해 각각의 그룹을 독립적으로 검증하고, 하나라도 실패했을 때만 최종 실패로 처리하도록 구조화했습니다.

### 배운 점과 인사이트

이번 작업의 핵심은 **"시스템의 상태 변화(State Transition)가 일어나는 찰나의 구간을 어떻게 정의할 것인가"**였습니다.

많은 개발자가 `if (date == today)` 식의 단순한 비교를 사용하지만, 실제 운영 환경에서는 데이터가 전이되는 '과도기(Transition Period)'가 반드시 존재합니다. 이 구간을 세밀하게 정의하지 않으면, 시스템은 논리적으로 맞음에도 불구하고 기술적으로는 '에러'라고 보고하게 됩니다.

**결국 좋은 모니터링이란 단순히 에러를 잡는 것이 아니라, '신뢰할 수 있는 신호(Trusted Signal)'만 남기는 과정**입니다. 노이즈를 제거함으로써 이제 저는 22시 이후에 울리는 알람이 '진짜 장애'라는 것을 확신할 수 있게 되었습니다.

작은 쉘 스크립트 수정이었지만, 시스템의 신뢰도를 높이는 엔지니어링의 본질을 다시 한번 생각하게 된 작업이었습니다.
