# [Dev Log] 데이터로 기록하는 성장: 개인 대시보드 업데이트와 개발 회고

안녕하세요, 씨피랩스의 CTO 송주한(Jeffrey)입니다.

엔지니어에게 있어 '기록'은 단순히 과거를 남기는 것이 아니라, 자신의 성장 궤적을 데이터로 확인하는 과정이라고 생각합니다. 저는 개인적으로 제가 어떤 프로젝트에 시간을 쏟고 있는지, 최근의 개발 흐름은 어떻게 변하고 있는지를 시각화하여 관리하는 대시보드를 운영하고 있습니다.

오늘은 이 대시보드의 데이터를 최신화하며, 최근 제가 집중하고 있는 기술적 고민들을 정리해 보았습니다.

### 🛠️ 오늘의 작업: 대시보드 데이터 업데이트

오늘 진행한 작업은 단순해 보이지만, 제 개발 생태계를 한눈에 보여주는 `dashboard.json` 파일의 업데이트였습니다. 

```json
// src/data/dashboard.json 변경 사항 중 일부
- "generated_at": "2026-05-31T23:00:01.819Z",
+ "generated_at": "2026-06-01T23:00:01.933Z",
```

단순한 날짜 변경 같지만, 이 데이터 뒤에는 최근 제가 씨피랩스에서 진행 중인 여러 프로젝트의 활동 내역이 담겨 있습니다. 특히 이번 업데이트에서 흥미로운 점은 제가 최근 집중하고 있는 리포지토리들의 변화입니다.

### 🔍 기술적 인사이트: 호환성과 사용자 경험(UX)

최근 커밋 로그를 살펴보면 제가 어떤 문제들을 풀고 있었는지 잘 나타납니다. 특히 두 가지 포인트가 눈에 띕니다.

**1. 레거시 환경과의 사투: Node.js 호환성 문제**
`meta-info-api` 프로젝트에서 다음과 같은 변경 사항이 있었습니다.
> `fix(node10-compat): replace AbortController with node-fetch v2 timeout option`

최신 표준인 `AbortController`를 사용하고 싶지만, 실제 운영 환경이나 특정 레거시 환경(Node 10 등)에서는 이를 지원하지 않는 경우가 많습니다. CTO로서 기술적 최신성을 유지하는 것도 중요하지만, **'실제로 동작하는 소프트웨어'**를 만드는 것이 최우선입니다. 때로는 최신 API 대신 라이브러리가 제공하는 옵션(`node-fetch v2 timeout`)으로 돌아가는 유연한 결정이 필요합니다.

**2. 디테일이 만드는 완성도: UI/UX의 한 끗 차이**
`meta-faucet` 프로젝트의 커밋 메시지에서도 제 고민이 보입니다.
> `ui(faucet-meta): natural wrapping in the network/RPC strip`

Web3 서비스, 특히 Faucet 같은 툴은 네트워크 주소나 RPC URL처럼 매우 긴 문자열을 다루는 경우가 많습니다. 이를 단순히 자르는 것이 아니라, 어떻게 '자연스럽게 줄바꿈(natural wrapping)' 하여 사용자에게 보여줄 것인가 하는 디테일이 서비스의 완성도를 결정짓습니다.

### 💡 마치며: 데이터가 주는 피드백

이번에 업데이트한 `activity` 배열의 숫자들을 보며, 제가 최근 어떤 리듬으로 개발하고 있는지 다시금 확인했습니다. 어떤 주는 특정 프로젝트에 몰입하고, 어떤 주는 여러 프로젝트를 오가며 병렬적으로 처리하죠.

20년 넘게 엔지니어로 살아오며 느낀 점은, **측정할 수 없는 것은 개선할 수 없다**는 것입니다. 코드 한 줄의 변경부터 프로젝트의 전체 흐름까지, 이렇게 데이터로 기록하고 회고하는 습관이 결국 더 나은 아키텍처와 더 견고한 제품을 만드는 밑거름이 된다고 믿습니다.

내일은 또 어떤 문제를 풀게 될까요? 기록하며 성장하는 개발자가 되겠습니다.
