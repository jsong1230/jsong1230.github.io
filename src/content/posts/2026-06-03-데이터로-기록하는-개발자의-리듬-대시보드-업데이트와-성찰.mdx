# [Dev Log] 데이터가 말해주는 개발자의 리듬: 대시보드 업데이트와 회고

안녕하세요, 송주한(Jeffrey Joo-Han Song)입니다.

엔지니어로 20년 넘게 커리어를 쌓아오며 제가 깨달은 것 중 하나는, 개발자의 성장과 생산성은 단순히 '얼마나 많은 코드를 짰느냐'가 아니라 '어떤 리듬으로 움직이고 있는가'에 달려 있다는 점입니다. 

오늘은 제 개인 대시보드의 데이터를 업데이트하며, 최근의 작업 흐름을 되짚어보는 시간을 가졌습니다. 단순한 `chore` 작업처럼 보일 수 있는 JSON 파일 수정이지만, 그 이면에는 현재 제가 집중하고 있는 기술적 고민들이 고스란히 담겨 있습니다.

### 오늘의 작업: 대시보드 데이터 동기화

오늘 진행한 작업은 `src/data/dashboard.json` 파일을 업데이트하여 최근의 깃허브 활동 내역을 반영하는 것이었습니다. 

```json
-  "generated_at": "2026-06-01T23:00:01.933Z",
+  "generated_at": "2026-06-02T23:00:01.580Z",
```

단순한 날짜 변경 같지만, 이 데이터가 보여주는 변화는 꽤 흥미롭습니다. 특히 `meta-info-api` 프로젝트의 변경 사항이 눈에 띕니다.

```json
+          "name": "meta-info-api",
+          "last_date": "2026-06-02",
+          "last_msg": "Merge branch 'main' into node10-compat",
+          "branch": "node10-compat",
```

### 기술적 결정과 인사이트: 하위 호환성의 딜레마

최근 제가 씨피랩스에서 진행하는 여러 프로젝트 중 하나인 `meta-info-api`에서는 `node10-compat` 브랜치를 통해 구버전 런타임과의 호환성을 맞추는 작업을 진행하고 있습니다. 

최신 스택을 사용하는 것은 즐거운 일이지만, 엔터프라이즈 환경이나 특정 레거시 인프라를 유지해야 하는 실무에서는 **'하위 호환성(Backward Compatibility)'**을 해결하는 것이 가장 까다롭고 중요한 문제입니다. 최신 JS 문법이나 라이브러리를 사용하고 싶어도, 대상 환경이 Node.js 10 버전이라면 `AbortController` 같은 최신 API를 사용할 수 없죠. 

이번 커밋 로그에 나타난 `Merge branch 'main' into node10-compat` 과정은 최신 기능의 이점을 가져오면서도, 구버전 환경에서 깨지지 않도록 하는 세밀한 조정 과정의 연속이었습니다. 이는 단순히 코드를 옮기는 작업이 아니라, 각 환경의 제약 사항을 정확히 이해하고 최적의 절충안(Trade-off)을 찾는 엔지니어링의 핵심 과정입니다.

### 마치며: 기록이 주는 힘

`jsong1230.github.io` 리포지토리의 커밋 수가 30일 기준 71회까지 올라간 것을 보며, 매일의 기록이 쌓여 만드는 힘을 다시금 느낍니다. 

블록체인, Web3, AI와 같이 빠르게 변하는 분야에서 일할수록, 내가 무엇을 고민했고 어떤 결정을 내렸는지 기록하는 습관은 매우 중요합니다. 훗날 비슷한 문제에 직면했을 때, 오늘의 이 JSON 데이터 한 줄과 커밋 메시지가 가장 확실한 가이드라인이 되어줄 것이기 때문입니다.

내일은 또 어떤 기술적 도전이 기다리고 있을까요? 씨피랩스에서의 여정과 함께 저의 기술적 성장 기록은 계속됩니다.
