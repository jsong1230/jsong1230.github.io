안녕하세요, 시니어 소프트웨어 엔지니어이자 코드 리뷰어입니다.

오늘의 작업 내역을 살펴보니 단순한 UI 수정부터 인프라 장애 대응(OAC 사고 우회), 백엔드 리소스 최적화, 모바일 클라이언트의 안정성 강화까지 **풀스택에 걸친 매우 폭넓고 밀도 높은 작업**이 이루어졌습니다. 특히 장애 상황에서 빠르게 우회 경로를 설계하고 이를 문서화하는 능력이 돋보입니다.

상세 리뷰를 전달드립니다.

---

## ✅ 잘한 점

1. **장애 대응의 신속성과 전략적 우회 (OAC $\rightarrow$ hls_proxy)**
   - CloudFront OAC 관련 이슈가 발생했을 때, 단순히 설정을 변경하는 것에 그치지 않고 `backend/app/routers/hls_proxy.py`를 통해 `boto3` 스트림으로 데이터를 전달하는 **프록시 계층을 빠르게 도입**하여 서비스 가용성을 확보한 결정이 매우 훌륭합니다. 인프라 레벨의 해결이 지연될 때 애플리케이션 레벨에서 우회로를 만드는 것은 시니어 엔지니어의 핵심 역량입니다.

2. **리소스 관리의 구체성 (LRU Cache 도입)**
   - `hls_builder.py`에서 `/tmp/audio_cache`에 대해 **3GB 캡을 둔 LRU cleanup 로직을 추가**한 점이 인상적입니다. 서버리스나 컨테이너 환경에서 `/tmp` 공간은 제한적이며, 이를 방치할 경우 `No space left on device` 에러로 이어져 서비스 전체가 다운될 수 있습니다. 이를 선제적으로 방어한 점을 높게 평가합니다.

3. **사용자 경험(UX)의 일관성 추구**
   - `Gnb.tsx`와 `ThemeToggle.tsx`에서 아이콘의 의미를 "현재 상태"가 아닌 **"다음 동작(Action)"으로 통일**한 점이 좋습니다. (light $\rightarrow$ MoonIcon). 이는 현대적인 UI 디자인 패턴이며, 사용자가 버튼을 눌렀을 때 어떤 변화가 일어날지 명확하게 예측하게 해줍니다.

4. **인프라 가시성 확보 (Network Architecture HTML)**
   - 복잡한 네트워크 구성(CF $\rightarrow$ ALB $\rightarrow$ ECS $\rightarrow$ S3/RDS)을 `mermaid.js`를 이용해 HTML 문서로 시각화한 점이 매우 좋습니다. 특히 `Oxblood Dial` 톤의 커스텀 테마까지 적용하여 가독성과 심미성을 모두 잡았으며, 이는 팀원 간의 커뮤니케이션 비용을 획기적으로 줄여줄 것입니다.

## 🔧 개선할 점

1. **프록시 계층의 성능 및 비용 모니터링 필요**
   - `hls_proxy.py`를 통해 S3 데이터를 백엔드 서버가 중계하게 되면, 기존 CF $\rightarrow$ S3 직접 통신보다 **백엔드 서버의 네트워크 I/O 및 CPU 부하가 증가**합니다. 
   - **제안:** 현재는 사고 우회를 위한 임시 조치이므로, 해당 엔드포인트에 대한 전용 메트릭(처리량, 응답 시간)을 모니터링하고, OAC 이슈 해결 후 빠르게 원래의 최적 경로로 복구하는 티켓을 발행하시길 권장합니다.

2. **하드코딩된 캐시 용량 설정의 외부화**
   - `hls_builder.py` 내의 `3GB cap` 설정이 코드 내에 상수로 존재할 가능성이 큽니다. 
   - **제안:** 인프라 환경(Dev/Prod)마다 디스크 용량이 다를 수 있으므로, 이 값을 `env-config.ts`나 환경 변수(`AUDIO_CACHE_MAX_SIZE_GB`)로 분리하여 인프라 설정 파일에서 관리하는 것이 더 유연합니다.

3. **모바일 재시도 로직의 지수 백오프(Exponential Backoff) 세부 설정**
   - `useHlsPlayer.native.ts`에 도입한 지수 백오프는 훌륭한 선택입니다. 다만, 최대 재시도 횟수(Max Retries)와 최대 대기 시간(Max Delay)이 적절히 설정되었는지 확인이 필요합니다. 
   - **제안:** 네트워크 장애가 장기화될 경우 무한히 재시도하는 것보다, 특정 횟수 이후 사용자에게 "네트워크 연결을 확인해주세요"라는 명확한 피드백을 주는 폴백(Fallback) 로직이 포함되어 있는지 점검하십시오.

## 💡 오늘의 기술 포인트

**"Resilience Engineering (회복 탄력성 설계)"**
오늘 작업의 핵심은 **'실패를 가정하고 대응하는 설계'**입니다. 
- **인프라 장애 $\rightarrow$ 애플리케이션 프록시 우회**
- **네트워크 불안정 $\rightarrow$ 지수 백오프 재시도**
- **디스크 풀(Full) $\rightarrow$ LRU 캐시 제한**

단순히 기능을 구현하는 것을 넘어, 시스템이 가질 수 있는 취약점을 파악하고 이를 보완하는 '방어적 프로그래밍'과 '인프라 유연성'을 동시에 보여준 수준 높은 작업이었습니다.
