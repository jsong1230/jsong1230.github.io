# [Dev Log] 데이터의 흐름으로 읽는 나의 성장 기록: 대시보드 업데이트와 자동화의 가치

안녕하세요, 송주한(Jeffrey Joo-Han Song)입니다.

엔지니어로 20년 넘게 커리어를 쌓아오며 제가 가장 중요하게 생각하는 것 중 하나는 '측정 가능한 성장'입니다. 단순히 "열심히 했다"는 느낌보다는, 내가 어떤 프로젝트에 얼마나 시간을 쏟았고, 어떤 리듬으로 코드를 작성했는지를 데이터로 확인하는 과정이 다음 단계로 나아가는 원동력이 되곤 합니다.

오늘은 개인 대시보드의 데이터를 업데이트하며, 최근의 작업 흐름을 되짚어보는 시간을 가졌습니다.

### 📊 단순한 업데이트, 그 이상의 의미

오늘 진행한 작업은 `src/data/dashboard.json` 파일을 업데이트하는 단순한 `chore` 작업이었습니다. 하지만 이 JSON 파일 하나에는 제가 씨피랩스에서 진행하는 다양한 프로젝트와 개인적인 연구 활동의 흔적이 고스란히 담겨 있습니다.

```json
// dashboard.json 변경 사항 중 일부
- "generated_at": "2026-06-10T23:00:02.033Z",
+ "generated_at": "2026-06-11T23:00:01.507Z",
...
+ {
+   "name": "metadium-explorer-stack",
+   "last_msg": "refactor: frontend-ws 로컬 수정을 patch 방식으로 전환...",
+   "branch": "master",
+   "activity": [0, 0, 0, 3, 1, 3, 0],
+ }
```

이번 업데이트에서 흥미로운 점은 `metadium-explorer-stack`과 같은 새로운 리포지토리가 대시보드에 추가되었다는 점입니다. 특히 `frontend-ws` 로컬 수정을 `patch` 방식으로 전환했다는 커밋 메시지에서 알 수 있듯, 외부 라이브러리나 공식 레포지토리의 코드를 수정해야 할 때 유지보수성을 어떻게 높일 것인가에 대한 기술적 고민이 반영된 결과물입니다.

### 🛠️ 기술적 결정: 왜 자동화된 대시보드인가?

많은 분이 "그냥 GitHub 잔디(Contribution Graph)를 보면 되지 않나?"라고 물으실 수 있습니다. 하지만 저는 더 세밀한 분석을 원했습니다.

1. **프로젝트별 가중치 분석**: 여러 리포지토리를 동시에 운영하는 CTO의 특성상, 특정 시점에 어떤 도메인(Web3, DID, AI 등)에 집중했는지 한눈에 파악해야 합니다.
2. **활동 리듬 파악**: `activity` 배열을 통해 일주일 단위의 커밋 빈도를 추적함으로써, 제가 번아웃에 가까운 과몰입 상태인지 혹은 적절한 페이스를 유지하고 있는지 체크합니다.
3. **데이터의 정량화**: `commits_7d`, `commits_30d`와 같은 지표를 통해 나의 생산성 추이를 정량적으로 관리합니다.

### 💡 인사이트: 기록은 기억을 지배한다

오늘의 작업은 아주 작은 코드 변경이었지만, 이를 통해 제가 최근 씨피랩스에서 진행 중인 여러 프로젝트의 진척도를 다시 한번 확인했습니다. 특히 `company-page`나 `cp-check-attendance`와 같은 실무 프로젝트들의 활동 수치가 변하는 것을 보며, 현재 팀의 리소스가 어디에 집중되고 있는지, 그리고 제가 기술적으로 서포트해야 할 지점이 어디인지 다시금 생각하게 되었습니다.

엔지니어에게 있어 '자동화'는 단순히 시간을 아끼는 도구가 아니라, **나의 상태를 객관적으로 바라보게 하는 거울**과 같습니다. 여러분도 자신만의 '성장 대시보드'를 만들어보시는 건 어떨까요? 

작은 기록들이 모여 결국 거대한 커리어를 만듭니다. 내일은 또 어떤 데이터가 저의 성장을 증명해줄지 기대됩니다.
