# [Dev Log] 데이터의 흐름으로 기록하는 나의 성장, 대시보드 업데이트

안녕하세요, 씨피랩스(CP Labs)의 CTO 송주한입니다.

엔지니어에게 있어 '기록'은 단순히 과거를 남기는 것이 아니라, 현재의 상태를 진단하고 미래의 방향성을 설정하는 중요한 데이터 포인트가 됩니다. 저는 개인적으로 제가 관리하는 여러 프로젝트와 레포지토리의 활동량을 시각화하여 관리하는 대시보드를 운영하고 있습니다.

오늘은 이 대시보드의 핵심 데이터 파일인 `dashboard.json`을 업데이트하며, 최근의 개발 흐름을 다시 한번 복기하는 시간을 가졌습니다.

### 🛠 오늘의 작업: 데이터 동기화와 활동 분석

오늘의 작업은 단순한 `chore` 성격의 데이터 업데이트처럼 보일 수 있지만, 그 내부를 들여다보면 제가 최근 어떤 기술적 고민에 집중하고 있었는지가 고스란히 드러납니다.

```json
- "last_msg": "Auto-generate weekly recap",
+ "last_msg": "Auto-generate daily posts",
```

가장 눈에 띄는 변경점은 `jsong1230.github.io` 레포지토리의 메시지 변화입니다. 기존의 '주간 리캡(Weekly Recap)' 자동 생성 방식에서 '일일 포스트(Daily Posts)' 생성 방식으로 파이프라인을 변경했습니다. 

**왜 이런 결정을 했을까요?**
기술의 변화 속도가 극심한 Web3와 AI 분야에서는 일주일이라는 시간조차 긴 공백이 될 수 있습니다. 매일의 작은 성취와 학습 내용을 파편화하여 기록하고, 이를 자동화된 파이프라인으로 관리함으로써 '지식의 휘발'을 막기 위함입니다.

### 💡 기술적 인사이트: 정량적 지표가 주는 의미

이번 업데이트에서 `activity` 배열의 수치 변화와 `commits_7d`, `commits_30d` 지표의 변동을 살펴보았습니다. 

특정 프로젝트의 커밋 수가 줄어들거나 활동 그래프가 변하는 것은 단순히 '일을 덜 했다'는 의미가 아닙니다. 이는 **리소스의 재배치**를 의미합니다. 예를 들어, 인프라 모니터링 스크립트(`monitor.sh`)에 텔레그램 알림을 추가하는 등의 안정화 작업이 완료되면, 자연스럽게 새로운 기능 개발이나 아키텍처 설계로 관심사가 이동하게 됩니다.

저는 씨피랩스에서 수많은 프로젝트를 리딩하며, 엔지니어의 생산성을 단순히 '커밋 횟수'로 판단하는 것이 얼마나 위험한지 경험했습니다. 진짜 중요한 것은 **"어떤 문제를 풀기 위해 어디에 집중하고 있는가"** 하는 맥락(Context)입니다.

### 📝 마치며

오늘의 `dashboard.json` 업데이트는 저에게 단순한 파일 수정이 아니라, 지난 며칠간의 몰입 지점을 확인하는 '회고'의 과정이었습니다. 

블록체인, DID, AI라는 거대한 흐름 속에서 길을 잃지 않는 방법은 결국 나만의 데이터를 쌓고, 그것을 객관적으로 바라보는 습관을 갖는 것이라고 생각합니다. 내일은 또 어떤 데이터가 제 대시보드를 채우게 될지 기대됩니다.
