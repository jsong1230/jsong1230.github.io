# [Dev Log] 사소한 설정과 문법 에러가 불러오는 '컨테이너 크래시 루프' 해결기

안녕하세요, 씨피랩스의 CTO 송주한(Jeffrey)입니다.

엔지니어로 20년 넘게 커리어를 쌓아오며 수많은 아키텍처를 설계하고 복잡한 비즈니스 로직을 구현했지만, 때로는 아주 사소한 '한 줄'의 설정이나 오타가 시스템 전체를 마비시키는 경험을 하곤 합니다. 오늘 제가 겪은 일이 바로 그랬습니다.

오늘은 씨피랩스에서 진행 중인 근태 관리 시스템(`cp-check-attendance`)의 빌드 및 런타임 이슈를 해결하며 느낀 점들을 공유해 보려 합니다.

### 1. Docker Compose: "당연히 알겠지"라는 생각의 위험성

먼저 맞닥뜨린 문제는 Docker Compose 빌드 실패였습니다. 프로젝트 구조상 여러 버전의 Dockerfile을 관리하고 있었는데, `docker-compose.yml`에서 빌드 컨텍스트만 지정하고 구체적인 Dockerfile 경로를 명시하지 않았습니다.

```yaml
# 수정 전
build:
  context: .

# 수정 후
build:
  context: .
  dockerfile: files/Dockerfile_v4.1
```

Docker는 기본적으로 루트 경로의 `Dockerfile`이라는 이름의 파일을 찾습니다. 하지만 저희 프로젝트는 버전 관리를 위해 `files/Dockerfile_v4.1`과 같이 경로와 이름을 명시적으로 구분해 사용하고 있었죠. "컨텍스트가 루트니까 알아서 찾겠지"라는 안일한 생각이 빌드 실패로 이어졌습니다. 인프라 설정에서는 **'명시적인 것이 암시적인 것보다 훨씬 낫다(Explicit is better than implicit)'**는 원칙을 다시 한번 되새겼습니다.

### 2. JavaScript의 함정: 하이픈(-) 하나가 만든 무한 루프

더 흥미로웠던(그리고 당혹스러웠던) 문제는 런타임 에러였습니다. 컨테이너가 뜨자마자 죽고, 다시 살아나고, 또 죽는 이른바 '크래시 루프(Crash Loop)'에 빠진 상황이었습니다. 로그를 분석해 보니 원인은 황당하게도 죽은 코드(Dead Code) 속에 숨어 있었습니다.

```javascript
// 문제의 코드
async function getAttendance-old(session,body){ ... }
```

과거에 사용하던 함수를 백업 용도로 남겨두면서 이름에 `getAttendance-old`라고 하이픈을 넣었는데, JavaScript에서 함수명에 하이픈은 허용되지 않습니다. 엔진은 이를 `getAttendance` 함수에서 `old`라는 변수를 빼기(`-`) 연산하려는 시도로 해석했고, 결국 `SyntaxError`를 발생시키며 프로세스를 종료시킨 것입니다.

현재는 `getAttendance`라는 최신 함수가 그 역할을 대체하고 있었기에, 문제가 된 구버전 코드를 과감히 주석 처리하여 해결했습니다.

### 3. 마치며: 엔지니어의 인사이트

오늘의 작업은 화려한 신기술 도입이나 거대한 아키텍처 변경은 아니었습니다. 하지만 두 가지 중요한 교훈을 얻었습니다.

첫째, **환경 설정의 명시성**입니다. 특히 팀 단위로 협업하는 프로젝트일수록 "누구나 당연히 알 것"이라고 생각하는 설정이 가장 위험합니다. 설정 파일은 그 자체로 완벽한 문서가 되어야 합니다.

둘째, **코드 정리의 중요성**입니다. "나중에 참고하려고" 남겨둔 주석 처리되지 않은 죽은 코드는 잠재적인 시한폭탄과 같습니다. 버전 관리 시스템(Git)이 있는데 굳이 코드 내에 `-old` 같은 접미사를 붙여 코드를 남길 필요가 없습니다. 필요하다면 Git History에서 찾으면 됩니다.

사소한 디테일이 시스템의 안정성을 결정짓는다는 점, 20년 차 엔지니어인 저에게도 매일 새롭게 다가오는 배움입니다. 내일은 더 견고한 코드로 찾아뵙겠습니다.
