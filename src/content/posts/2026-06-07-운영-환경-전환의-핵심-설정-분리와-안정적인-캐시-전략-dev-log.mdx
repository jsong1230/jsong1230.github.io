# [Dev Log] 운영 전환의 디테일: 설정의 분리와 캐시 전략

안녕하세요, 송주한(Jeffrey Joo-Han Song)입니다.

개발자에게 가장 긴장되는 순간 중 하나는 로컬이나 스테이징 환경에서 잘 돌아가던 서비스가 실제 운영(Production) 환경으로 넘어가는 찰나일 것입니다. 단순한 배포를 넘어, '운영 환경'이라는 특수성이 주는 제약과 리스크를 어떻게 관리하느냐가 엔지니어의 역량을 결정짓곤 합니다.

오늘은 씨피랩스(CP Labs)의 컴퍼니 페이지를 운영 환경으로 전환하기 위한 최종 준비 작업을 진행했습니다. 단순한 기능 구현보다는 **안정성, 보안, 그리고 SEO(검색 엔진 최적화)**라는 세 가지 관점에서 시스템을 정비하는 데 집중했습니다.

### 1. 시크릿 관리의 정석: `.env` 분리와 템플릿화

가장 먼저 손을 댄 곳은 환경 변수 관리입니다. 개발 초기에는 편의상 `.env.local` 같은 파일에 시크릿을 두고 관리하곤 하지만, 이는 보안 사고의 주범이 됩니다.

이번 작업에서 저는 `.env.local`을 완전히 삭제하고 `.gitignore`에 추가하여 커밋 히스토리에 시크릿이 남지 않도록 조치했습니다. 대신 `.env.example` 파일을 생성해 팀원들이 어떤 설정값이 필요한지 한눈에 알 수 있게 했습니다.

```diff
- .env.local           | 18 ----------
+ .env.example         | 22 +++++++++++++
```

특히 `docker-compose.yml`에서 환경 변수를 직접 주입하는 방식 대신 `env_file: .env` 형식을 채택하여, 인프라 설정과 실제 비밀값의 결합도를 낮췄습니다. 이미 히스토리에 남은 시크릿들은 운영 전환 전 로테이션(Rotation) 작업을 거칠 예정이며, 이를 위해 `docs/PRODUCTION.md`에 상세 체크리스트를 작성해 두었습니다.

### 2. 사용자 경험을 결정짓는 캐시 제어 (Cache-Control)

Next.js 기반의 프로젝트를 운영할 때 흔히 겪는 문제 중 하나가 **재배포 후 구버전 캐시로 인한 Hydration 오류**입니다. 서버에서 내려준 HTML과 클라이언트의 JS 버전이 맞지 않아 화면이 깜빡이거나 에러가 발생하는 현상이죠.

이를 방지하기 위해 `next.config.js`에서 HTML 및 데이터 응답에 `Cache-Control: no-cache` 헤더를 설정했습니다. 이는 브라우저가 캐시를 저장하되, 사용하기 전에 항상 서버에 유효성 검사를 요청하게 함으로써 최신 버전을 안전하게 서빙하기 위한 결정입니다.

### 3. 검색 엔진을 위한 기본기: Sitemap과 Robots.ts

B2B 서비스의 얼굴인 컴퍼니 페이지에서 SEO는 필수입니다. 기존의 정적 `robots.txt` 파일을 제거하고, Next.js의 `app/robots.ts`와 `app/sitemap.ts`를 통해 동적으로 생성하도록 변경했습니다.

```typescript
// app/robots.ts
export default function robots(): MetadataRoute.Robots {
  return {
    rules: {
      userAgent: '*',
      allow: '/',
      disallow: ['/admin/', '/api/'], // 관리자 및 API 경로 노출 차단
    },
    sitemap: `${BASE_URL}/sitemap.xml`,
  };
}
```

단순히 파일을 만드는 것에 그치지 않고, `NEXT_PUBLIC_BASE_URL` 환경 변수를 통해 도메인 변경에 유연하게 대응하도록 설계했습니다. 특히 `/admin`이나 `/api` 경로를 `disallow` 처리하여 불필요한 인덱싱을 막고 보안성을 높였습니다.

### 마치며: '작은 설정'이 만드는 '큰 차이'

오늘 작업한 내용은 화려한 알고리즘이나 복잡한 아키텍처 설계는 아닙니다. 하지만 시크릿 분리, 캐시 헤더 설정, SEO 최적화 같은 디테일들이 모여 서비스의 '완성도'를 결정합니다. 

20년 넘게 엔지니어로 일하며 배운 점이 있다면, **"운영 환경에서의 1%의 불확실성을 제거하는 것이 99%의 기능 구현보다 중요하다"**는 것입니다. 씨피랩스의 서비스가 사용자들에게 끊김 없이, 그리고 안전하게 전달될 수 있도록 마지막까지 꼼꼼하게 챙기겠습니다.
