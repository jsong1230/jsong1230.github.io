# [Dev Log] 데이터의 흐름으로 읽는 나의 개발 리듬: 대시보드 업데이트와 성찰

안녕하세요, 송주한(Jeffrey Joo-Han Song)입니다.

20년 넘게 엔지니어로 일하며 수많은 시스템을 설계하고 구축해왔지만, 제가 여전히 가장 중요하게 생각하는 것 중 하나는 바로 **'자기 객관화'**입니다. 내가 어디에 시간을 쓰고 있는지, 어떤 프로젝트에 몰입하고 있는지를 정량적인 데이터로 확인하는 과정은 더 나은 생산성을 위한 필수적인 단계라고 믿습니다.

오늘은 개인 대시보드의 핵심 데이터 파일인 `dashboard.json`을 업데이트했습니다. 단순한 데이터 갱신처럼 보일 수 있지만, 이 과정에서 제가 바라보는 '개발 리듬'에 대한 인사이트를 공유하고자 합니다.

### 🛠️ 오늘의 작업: 대시보드 데이터 동기화

오늘 진행한 작업은 간단합니다. 최근 7일 및 30일간의 커밋 활동량과 마지막 커밋 메시지 등을 최신화하는 작업이었습니다. 

```json
// src/data/dashboard.json 변경 사항 중 일부
-  "generated_at": "2026-05-21T23:00:01.757Z",
+  "generated_at": "2026-05-22T23:00:01.436Z",
```

단순히 날짜가 바뀌는 것이 아니라, 제가 관리하는 여러 저장소(`heritageradio`, `jsong1230.github.io`, `thepol-dev` 등)의 활동 지표가 실시간으로 반영되는 과정입니다.

### 💡 기술적 결정과 인사이트: 왜 '데이터'로 기록하는가?

저는 씨피랩스에서 CTO로서 복잡한 Web3, DID, AI 프로젝트들을 리딩하며 '가시성(Visibility)'의 중요성을 뼈저리게 느꼈습니다. 팀의 진척도를 파악하기 위해 지표를 설정하듯, 개인의 성장 또한 지표화할 필요가 있습니다.

이번 업데이트에서 흥미로운 점은 저장소별 `activity` 배열의 변화입니다. 예를 들어 `heritageradio` 저장소의 활동량을 보면:

```json
"activity": [
  4,
  3,
  0,
  5,
  1,
  0,
  0
],
"commits_30d": 803
```

한 달 동안 800회가 넘는 커밋이 발생했다는 것은 해당 프로젝트가 현재 매우 치열한 개발 단계(Iteration)에 있음을 보여줍니다. 반면, `thepol-dev`나 `trump-code` 같은 프로젝트는 상대적으로 낮은 빈도를 유지하며 유지보수 단계에 있음을 알 수 있죠.

이렇게 데이터를 통해 내 개발 패턴을 분석하면 다음과 같은 결정을 내릴 수 있습니다.
1. **에너지 분배:** 특정 프로젝트에 과하게 몰입해 다른 중요한 실험(AI, Web3 연구 등)을 놓치고 있지는 않은가?
2. **리듬 파악:** 내가 주로 어느 요일에 집중력이 높은가? (Activity 배열의 패턴 분석)
3. **성취감의 정량화:** 막연하게 "열심히 했다"가 아니라 "이번 달에 800번의 개선을 이루어냈다"는 확신을 얻습니다.

### 🚀 마치며

많은 개발자가 코드 작성 그 자체에 집중하지만, 시니어 엔지니어이자 아키텍트로 성장하기 위해서는 **'자신을 관리하는 시스템'**을 구축하는 것이 중요합니다. 

오늘의 `chore: Update dashboard data` 커밋은 비록 비즈니스 로직을 변경하는 거창한 작업은 아니었지만, 저에게는 다음 한 주를 계획하기 위한 이정표를 세우는 작업이었습니다. 

여러분은 자신의 성장 궤적을 어떻게 기록하고 계신가요? 때로는 단순한 JSON 파일 하나가 그 어떤 복잡한 분석 도구보다 더 명확한 답을 주기도 합니다.
