# [Dev Log] 데이터의 흐름으로 기록하는 나의 성장, 대시보드 업데이트의 의미

안녕하세요, 씨피랩스의 CTO 송주한(Jeffrey)입니다.

엔지니어로서 가장 즐거운 순간 중 하나는 내가 구축한 시스템이 자동으로 작동하며 나의 활동을 기록하고, 그것이 시각화되어 나타나는 것을 볼 때입니다. 오늘은 제 개인 프로젝트인 `gram-jsong` 대시보드의 데이터를 업데이트하며, '기록의 자동화'와 '데이터 기반의 회고'에 대해 짧게 생각해보는 시간을 가졌습니다.

### 오늘의 작업: 대시보드 데이터 갱신

오늘 진행한 작업은 단순해 보이지만, 제 개발 생태계를 한눈에 보여주는 `dashboard.json` 파일을 업데이트하는 것이었습니다. 

```json
-  "generated_at": "2026-07-06T23:00:01.672Z",
+  "generated_at": "2026-07-07T23:00:01.707Z",
```

단순한 날짜 변경처럼 보이지만, 이 데이터는 제가 관리하는 여러 저장소(Repository)의 커밋 횟수, 최근 활동 메시지, 그리고 7일/30일간의 활동 지표를 자동으로 수집하여 반영한 결과물입니다.

### 기술적 결정과 인사이트: 왜 자동화된 대시보드인가?

20년 넘게 엔지니어로 일하며 제가 깨달은 점은, **"측정되지 않는 것은 개선될 수 없다"**는 것입니다. 특히 Web3나 AI처럼 변화 속도가 빠른 분야에서 활동하는 전문가라면, 자신이 어디에 시간을 쏟고 있는지 객관적으로 파악하는 것이 중요합니다.

이번 업데이트 내역을 보면 흥미로운 점이 있습니다. `claude-agent` 저장소의 활동이 업데이트되었고, `heritageradio`나 `metadium_homepage` 같은 프로젝트들의 우선순위가 데이터상에서 자연스럽게 재배치되었습니다.

```diff
-          "name": "heritageradio",
-          "last_date": "2026-07-03",
+          "name": "claude-agent",
+          "last_date": "2026-07-07",
```

단순히 `chore` 작업으로 분류되는 데이터 업데이트지만, 이를 통해 저는 다음과 같은 인사이트를 얻습니다.

1. **관심사의 이동 파악**: 최근 제가 AI 에이전트(`claude-agent`) 구현과 로그 자동화에 더 많은 에너지를 쓰고 있음을 수치로 확인했습니다.
2. **지속 가능성 확인**: `commits_30d` 지표를 통해 특정 프로젝트에 매몰되지 않고 여러 도메인(DID, AI, Web3)에 걸쳐 균형 있게 활동하고 있는지 체크합니다.
3. **자동화의 가치**: 이 모든 과정이 수동 기록이 아닌, 스크립트에 의해 생성된다는 점이 핵심입니다. 기록을 위한 기록에 시간을 쓰는 것이 아니라, 기록된 데이터를 '해석'하는 데 시간을 쓰게 만드는 설계입니다.

### 마치며

씨피랩스에서 CTO로서 복잡한 아키텍처를 설계하고 팀을 리딩하는 일도 중요하지만, 이렇게 개인적인 개발 루틴을 자동화하고 관리하는 과정에서 얻는 즐거움이 저를 계속 성장하게 만드는 동력이 됩니다.

작은 커밋 하나, JSON 파일의 숫자 하나가 모여 결국 한 엔지니어의 히스토리가 됩니다. 여러분은 자신의 성장을 어떻게 측정하고 계신가요?
