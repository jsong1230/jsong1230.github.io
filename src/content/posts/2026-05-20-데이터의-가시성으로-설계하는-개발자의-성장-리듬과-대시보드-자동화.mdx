# [Dev Log] 데이터의 흐름으로 기록하는 성장: 대시보드 자동화와 개발 리듬

안녕하세요, 씨피랩스의 CTO 송주한(Jeffrey)입니다.

엔지니어로서 20년 넘게 커리어를 쌓아오며 제가 가장 중요하게 생각하는 것 중 하나는 '가시성(Visibility)'입니다. 내가 어떤 고민을 했고, 어디에 시간을 쏟았으며, 기술적 성장이 어느 방향으로 흐르고 있는지를 데이터로 확인하는 과정은 단순한 기록 이상의 의미가 있습니다. 그것은 곧 스스로의 개발 리듬을 파악하고 최적화하는 과정이기 때문입니다.

오늘은 제 개인 대시보드(`jsong1230.github.io`)의 데이터를 업데이트하며, 최근의 작업 흐름을 되짚어 보았습니다.

### 단순한 데이터 업데이트, 그 이상의 의미

오늘의 커밋은 겉보기에 단순한 `chore: Update dashboard data`입니다. `dashboard.json` 파일의 수치를 변경하는 작업이죠. 하지만 이 JSON 파일에 담긴 숫자들은 제가 최근 집중하고 있는 기술적 관심사와 씨피랩스에서 추구하는 엔지니어링 방향성을 그대로 투영하고 있습니다.

```json
// diff 중 흥미로운 부분
- "name": "thepol-dev",
- "last_msg": "Initial thepol dev infra (docker-compose + nginx + patches + README)",
+ "name": "thepol-dev",
+ "last_msg": "Move compose build contexts to external/ vendored sources",
```

특히 눈에 띄는 부분은 `thepol-dev` 저장소의 변경 사항입니다. 초기 인프라 설정(`Initial thepol dev infra`) 단계에서 한 걸음 더 나아가, 빌드 컨텍스트를 외부/벤더 소스로 분리(`Move compose build contexts to external/ vendored sources`)하는 작업을 진행했습니다.

### 기술적 결정: 왜 빌드 컨텍스트를 분리했는가?

인프라를 설계할 때 제가 항상 고민하는 지점은 **'결합도(Coupling)'**와 **'재사용성'**입니다. 모든 빌드 파일을 하나의 컨텍스트에 몰아넣으면 초기 설정은 빠를지 모르나, 프로젝트가 커질수록 빌드 캐시 효율이 떨어지고 의존성 관리가 복잡해집니다.

이번에 빌드 컨텍스트를 외부 소스로 분리한 결정은 다음과 같은 인사이트를 바탕으로 했습니다.

1. **빌드 최적화**: 불필요한 파일이 빌드 컨텍스트에 포함되어 Docker 데몬으로 전송되는 시간을 줄이고, 레이어 캐싱을 극대화하기 위함입니다.
2. **모듈화**: 공통으로 사용되는 벤더 소스를 분리함으로써, 향후 다른 서비스로 확장할 때 동일한 기반 설정을 쉽게 재사용할 수 있도록 구조화했습니다.

### 기록이 주는 인사이트

대시보드의 `activity` 배열과 `commits_7d` 수치를 보면, 최근 `heritageradio`와 `thepol-dev` 사이에서 균형을 잡으며 작업하고 있음을 알 수 있습니다. 특히 `commits_30d` 수치가 높게 유지되는 것은 단순한 양적 팽창이 아니라, 지속적인 리팩토링과 작은 개선(Continuous Improvement)을 반복하고 있다는 증거이기도 합니다.

많은 이들이 "데이터 업데이트 같은 단순 작업에 왜 시간을 쓰는가"라고 물을 수 있습니다. 하지만 저는 믿습니다. **측정할 수 없다면 개선할 수 없고, 기록하지 않는다면 기억되지 않습니다.**

씨피랩스에서도 저는 팀원들이 자신의 성장을 데이터로 확인하고, 기술적 의사결정의 근거를 기록하는 문화를 만들기 위해 노력하고 있습니다. 오늘 업데이트한 이 작은 JSON 파일 하나가, 결국 더 견고한 아키텍처와 효율적인 개발 프로세스로 이어지는 밑거름이 될 것입니다.

오늘 여러분의 대시보드에는 어떤 성장의 기록이 남았나요?
