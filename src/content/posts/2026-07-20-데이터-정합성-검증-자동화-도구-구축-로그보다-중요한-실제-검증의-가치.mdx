# [Dev Log] 데이터 정합성 검증 도구 구축과 원격 작업의 뼈아픈 교훈

안녕하세요, 씨피랩스의 CTO 송주한입니다.

최근 진행 중인 '헤리티지 라디오' 프로젝트에서 과거의 기록을 복원하는 작업을 이어가고 있습니다. 오늘은 특히 <대한뉴스(NEWS_ERA)> 콘텐츠 100편(w13~w16 분량)을 생성하고, 이를 검증하는 자동화 도구를 구축하는 작업을 진행했습니다.

단순한 데이터 생성처럼 보이지만, 그 과정에서 마주한 기술적 이슈와 운영상의 실수, 그리고 이를 해결하며 얻은 인사이트를 공유하고자 합니다.

### 1. 문제는 '믿음'이 아니라 '검증'에 있다

대량의 오디오 파일을 생성하고 DB에 등록할 때, 가장 위험한 것이 "로그에 에러가 없으니 잘 되었겠지"라는 믿음입니다. 특히 FFmpeg을 이용한 concat(병합) 작업과 `loudnorm` 필터를 통한 라우드니스 정규화 과정에서는 파일이 생성되었더라도 실제 재생 시간이 짧거나, 무음 구간이 발생하는 등의 '논리적 결함'이 생길 수 있습니다.

이를 위해 `scripts/verify_news_era.py`라는 검증 도구를 신설했습니다. 단순히 파일 존재 여부만 체크하는 것이 아니라, 다음과 같은 다각도 검증 로직을 구현했습니다.

*   **DB ↔ Disk 정합성:** DB에 기록된 경로에 실제 파일이 존재하는가?
*   **정밀 재생 시간 측정:** `ffprobe`를 통해 실제 duration이 기준치(예: 600s)에 부합하는가?
*   **오디오 품질 스팟 체크:** `volumedetect` 필터를 통해 `mean_volume`을 측정하여 무음 파일이나 비정상적인 볼륨 레벨을 탐지합니다.

```python
def mean_volume(p: Path):
    """volumedetect mean_volume(dB) — 근사 무음 탐지"""
    try:
        r = subprocess.run(["ffmpeg", "-i", str(p), "-af", "volumedetect", "-f", "null", "-"],
                           capture_output=True, text=True, timeout=60)
        for line in r.stderr.splitlines():
            if "mean_volume:" in line:
                return float(line.split("mean_volume:")[1].split("dB")[0].strip())
    except Exception:
        pass
    return None
```

실제로 검증 과정에서 한 파일이 600초가 아닌 557초로 측정되는 `dur_mismatch` 사례를 발견했습니다. 분석 결과, `loudnorm` 작업이 파일에 기록되는 시점과 검증 스크립트가 읽는 시점이 겹치며 발생한 일시적인 transient 현상임을 규명했고, settle 후 재확인을 통해 정합성을 맞췄습니다.

### 2. 운영 교훈: SSH 세션의 배신과 `nohup`의 중요성

오늘 작업 중 가장 뼈아픈 실수는 운영 환경에서의 '세션 관리'였습니다. 

장시간 소요되는 생성 작업을 `docker compose exec`로 실행하고 백그라운드로 돌렸지만, SSH 연결이 끊어지자 원격 프로세스가 `SIGHUP` 시그널을 받고 24편째에서 갑자기 사망했습니다. 로그는 0바이트였고 DB 커밋은 이루어지지 않은 상태였죠.

**교훈: 장시간 실행되는 원격 잡(Remote Job)은 반드시 SSH 라이프타임과 분리해야 합니다.**

이후 다음과 같이 `nohup`과 쉘 실행 구조를 변경하여 재시도했습니다.
`nohup sh -c "... > log; echo DONE" &`

이렇게 detach 시킨 후에는 짧은 poll(로그 확인)로만 감시함으로써, 제 로컬 PC의 네트워크 상태와 상관없이 서버에서 작업이 완주될 수 있도록 조치했습니다. 기초적인 부분이지만, 대규모 데이터 처리 시 간과하기 쉬운 실무적인 포인트였습니다.

### 마치며

엔지니어로서 가장 희열을 느끼는 순간은 '완벽하게 자동화된 시스템'을 만들었을 때가 아니라, '내 시스템이 틀렸음을 증명하는 검증 도구'를 통해 결함을 찾아내고 수정했을 때인 것 같습니다. 

데이터의 양이 많아질수록 신뢰는 코드가 아닌 '검증 결과'에서 나옵니다. 내일은 오늘 구축한 검증 도구를 더 고도화하여, 사람이 개입하지 않아도 품질이 보장되는 파이프라인을 완성해 보겠습니다.
