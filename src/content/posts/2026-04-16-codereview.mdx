안녕하세요. 시니어 소프트웨어 엔지니어이자 코드 리뷰어입니다.

제출해주신 `heritageradio` 프로젝트의 v2.0 Phase 1 MVP 커밋을 분석했습니다. 이번 작업은 단순한 기능 추가를 넘어 **TTS 품질 고도화, 사용자 경험(UX) 개선, 데이터 파이프라인 확장**이라는 세 마리 토끼를 동시에 잡으려는 매우 밀도 높은 작업으로 보입니다. 특히 오디오 도메인에서 발생할 수 있는 엣지 케이스(환각, 볼륨 불균형)를 기술적으로 해결하려는 접근이 인상적입니다.

상세 리뷰 내용은 다음과 같습니다.

---

## ✅ 잘한 점

**1. 도메인 특화된 TTS 정규화 및 후처리 파이프라인 구축**
- `backend/agents/text_normalize.py`를 통해 숫자, 영문, 한자를 한글 발음으로 변환하는 정규화 단계를 도입한 점이 매우 훌륭합니다. TTS 엔진의 성능에만 의존하지 않고, 전처리(Normalization) $\rightarrow$ 생성 $\rightarrow$ 후처리(Post-processing)의 전체 파이프라인을 설계하여 출력 품질의 일관성을 확보했습니다.
- 특히 `babbling cutoff` (silence-detect 기반 hard cut) 도입은 LLM 기반 TTS에서 흔히 발생하는 '끝맺음 실패 및 환각' 문제를 휴리스틱하게 해결한 실무적인 최적화 사례입니다.

**2. 철저한 캐시 제어 및 실시간성 확보**
- `dynamic="force-dynamic"` 설정과 `Cache-Control: no-store` 헤더를 통해 API 응답의 캐싱을 명시적으로 방지한 결정이 좋습니다. 에피소드 생성 상태가 빈번하게 변하는 서비스 특성상, 클라이언트가 최신 상태(`ready` 여부)를 즉각적으로 반영하게 함으로써 UX 저하를 막았습니다.

**3. 데이터 기반의 품질 보강 전략**
- `event_filler.py`를 통해 부족한 역사적 사건 데이터를 Gemma4로 보강하고, 이를 `sources=['gemma4-generated']`로 태깅하여 데이터의 출처를 명확히 한 점이 좋습니다. 이는 추후 데이터 검증(Audit) 단계에서 생성 데이터와 실제 데이터를 구분하여 관리할 수 있게 해주는 좋은 설계입니다.

**4. 세심한 브랜드 경험(BX) 설계**
- `scripts/make_favicon.py`를 통해 빈티지 라디오 컨셉의 파비콘을 코드로 생성하고, iOS safe-area 및 모바일 뷰포트 최적화를 진행한 점은 개발자가 제품의 최종 사용자 경험까지 깊게 고민하고 있음을 보여줍니다.

---

## 🔧 개선할 점

**1. 하드코딩된 매핑 및 설정값의 외부 설정화**
- `tts_client.VOICE_REFS`와 `episode_generator.VOICE_REFS_BY_DECADE` 간의 동기화 문제가 발생했던 점을 미루어 볼 때, 현재 보이스 매핑 정보가 코드 내에 분산되어 있는 것으로 보입니다.
- **개선 제안:** 보이스 설정, 연대별 매핑, TTS 파라미터(LUFS 값, silence 길이 등)를 `config.yaml`이나 `settings.json` 같은 외부 설정 파일로 통합 관리하여, 코드 수정 없이 설정만으로 튜닝이 가능하도록 구조화하시길 권장합니다.

**2. 에러 핸들링의 구체화 (TTS 게이트웨이 우회 관련)**
- 503 Timeout 문제를 해결하기 위해 9005 포트로 직접 우회하는 방식을 택하셨습니다. 이는 빠른 해결책(Quick-fix)으로는 훌륭하지만, 인프라 관점에서는 게이트웨이의 역할(로드밸런싱, 인증, 로깅)을 포기하는 결정입니다.
- **개선 제안:** 임시 조치임을 명시하고, 게이트웨이의 `stale connection` 원인을 분석하여 Keep-alive 설정이나 Timeout 설정을 조정하는 근본적인 인프라 개선 티켓을 생성하시길 권장합니다.

**3. 프롬프트 의존성 관리**
- `news_script_generator.py`의 시스템 프롬프트에 "TTS 출력용 표기 규칙"을 강제하셨습니다. 하지만 LLM은 항상 규칙을 100% 준수하지 않습니다.
- **개선 제안:** 프롬프트에 의존하기보다, 이미 구현하신 `text_normalize.py`를 **최종 출력 직전의 필수 파이프라인**으로 배치하여, LLM이 어떤 형태로 출력하든 정규화 모듈이 이를 보정하도록 보장하는 구조(Guardrail)를 강화하십시오.

## 💡 오늘의 기술 포인트

**1. 오디오 정규화 파이프라인 (Audio Normalization Pipeline)**
- `silence trim` $\rightarrow$ `loudnorm` $\rightarrow$ `silence insertion` $\rightarrow$ `babbling cutoff`로 이어지는 후처리 공정은 오디오 서비스의 상용 수준 품질을 결정짓는 핵심 기술 포인트입니다. 특히 LUFS(Loudness Units relative to Full Scale) 기준의 볼륨 균일화는 청취자의 피로도를 줄이는 매우 전문적인 접근입니다.

**2. 하이브리드 데이터 보강 (Hybrid Data Augmentation)**
- 실제 역사 데이터와 LLM 생성 데이터를 혼합하여 데이터셋의 밀도를 높이되, 메타데이터(`sources`)를 통해 추적 가능성을 확보한 패턴은 데이터 엔지니어링 측면에서 매우 효율적인 전략입니다.
