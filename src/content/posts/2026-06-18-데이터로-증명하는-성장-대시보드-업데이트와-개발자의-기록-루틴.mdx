# [Dev Log] 데이터로 기록하는 성장: 대시보드 업데이트와 개발자의 루틴

안녕하세요, 씨피랩스(CP Labs)의 CTO 송주한입니다.

엔지니어에게 있어 '기록'은 단순히 과거를 남기는 일이 아니라, 현재의 속도를 측정하고 미래의 방향을 결정하는 중요한 지표가 됩니다. 저는 개인적으로 제가 관리하는 여러 프로젝트의 활동량을 시각화하여 보여주는 대시보드를 운영하고 있습니다. 오늘은 이 대시보드의 데이터를 업데이트하며 느낀 몇 가지 생각들을 공유해보려 합니다.

### 오늘의 작업: 대시보드 데이터 동기화

오늘 진행한 작업은 단순해 보이지만 저에게는 매우 중요한 루틴인 `dashboard.json` 업데이트였습니다. 제가 진행 중인 다양한 레포지토리의 최근 커밋 메시지, 활동 주기, 그리고 커밋 횟수를 갱신하는 작업이죠.

이번 업데이트에서 가장 눈에 띄는 변화는 씨피랩스의 홈페이지(`cplabs_homepage`) 관련 작업량이 대폭 반영된 점입니다.

```json
{
  "name": "cplabs_homepage",
  "last_date": "2026-06-17",
  "last_msg": "fix(header): 언어 전환 시 현재 경로 유지 (KO→EN이 메인으로 점",
  "branch": "master",
  "activity": [0, 0, 0, 11, 36, 59, 0],
  "commits_7d": 106,
  "commits_30d": 106
}
```

특히 `commits_7d`가 106건에 달하는 것을 보며, 최근 씨피랩스의 브랜드 아이덴티티를 정립하고 사용자 경험(UX)을 개선하기 위해 얼마나 치열하게 고민했는지가 데이터로 증명되는 기분이었습니다. 특히 다국어 지원 시 현재 경로를 유지하는 디테일한 수정 사항(`fix(header): 언어 전환 시 현재 경로 유지`) 같은 부분은 사용자에게는 작게 느껴질 수 있지만, 제품의 완성도를 결정짓는 핵심적인 '마지막 1%'의 작업입니다.

### 기술적 결정과 인사이트: 왜 데이터로 관리하는가?

단순히 GitHub의 잔디(Contribution Graph)를 보는 것만으로도 충분할 수 있습니다. 하지만 저는 굳이 JSON 형태로 데이터를 추출해 별도의 대시보드로 관리합니다. 여기에는 두 가지 이유가 있습니다.

**첫째, 맥락의 가시화입니다.**
단순한 점(dot)이 아니라 `last_msg`를 함께 기록함으로써, 내가 특정 시점에 어떤 기술적 문제에 집중했는지 빠르게 회고할 수 있습니다. 예를 들어 `heritageradio` 프로젝트에서 `NEWS_ERA` 보강 작업을 진행한 흐름을 한눈에 파악하는 식입니다.

**둘째, 메타 인지의 강화입니다.**
`activity` 배열을 통해 일주일 단위의 리듬을 확인합니다. 특정 프로젝트에 과몰입하고 있지는 않은지, 혹은 방치되고 있는 중요한 태스크는 없는지 객관적으로 바라볼 수 있게 합니다.

### 마치며

20년 넘게 엔지니어로 살아오며 깨달은 점은, 뛰어난 개발자는 코드만 잘 짜는 사람이 아니라 **자신의 생산성을 관리하고 최적화할 줄 아는 사람**이라는 것입니다. 

오늘의 `chore: Update dashboard data` 커밋은 기능적인 변경점은 없지만, 저라는 '엔지니어'라는 시스템을 최적화하기 위한 일종의 헬스체크와 같습니다. 여러분도 자신만의 성장 지표를 데이터로 기록해 보시는 건 어떨까요?
