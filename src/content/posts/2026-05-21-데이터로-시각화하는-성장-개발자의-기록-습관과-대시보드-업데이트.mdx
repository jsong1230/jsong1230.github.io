# [Dev Log] 데이터가 말해주는 성장: 대시보드 업데이트와 개발자의 기록 습관

안녕하세요, 씨피랩스(CP Labs)의 CTO 송주한입니다.

개발자에게 있어 '기록'은 단순히 과거를 남기는 일이 아닙니다. 내가 어떤 고민을 했고, 어떤 리듬으로 코드를 작성했으며, 어디에 시간을 쏟았는지를 시각화하는 과정은 일종의 '자기 객관화'와 같습니다. 오늘은 제가 개인적으로 운영 중인 개발 활동 대시보드(`jsong1230.github.io`)의 데이터를 업데이트하며 느낀 몇 가지 인사이트를 공유하려 합니다.

### 오늘의 작업: 대시보드 데이터 동기화

오늘 진행한 작업은 단순해 보이지만 중요한 루틴인 `dashboard.json` 업데이트였습니다. 이 파일은 제가 참여하고 있는 여러 프로젝트의 커밋 로그, 활동량, 최근 메시지 등을 집계하여 시각화하는 데이터 소스입니다.

```json
// src/data/dashboard.json 변경 사항 중 일부
+ {
+   "name": "heritageradio",
+   "last_date": "2026-05-20",
+   "last_msg": "[landing+main] PostHog 도입 검토 brief — 자체 인프라 vs PostHog",
+   "branch": "dev",
+   "commits_7d": 57,
+   "commits_30d": 897
+ }
```

### 기술적 결정과 고민: '무엇을 기록할 것인가'

단순한 JSON 업데이트처럼 보이지만, 이 대시보드의 구조를 설계하며 제가 중요하게 생각한 점은 **'추세(Trend)'와 '맥락(Context)'**입니다.

1. **활동량의 시각화 (`activity` 배열):** 단순히 총 커밋 수를 기록하는 것이 아니라, 최근 7일간의 일별 활동량을 배열로 관리합니다. 이를 통해 제가 현재 어떤 프로젝트에 몰입하고 있는지, 혹은 번아웃이 오지는 않았는지 리듬을 체크합니다.
2. **최근 메시지의 보존 (`last_msg`):** 커밋 메시지는 당시의 고민을 가장 잘 담고 있는 캡슐입니다. 예를 들어 위 코드에서 `PostHog 도입 검토`라는 메시지를 통해, 최근 제가 단순 개발을 넘어 '데이터 분석 인프라의 효율성(자체 구축 vs SaaS)'이라는 아키텍처적 고민을 하고 있음을 한눈에 알 수 있습니다.

### 코드 변경을 통해 얻은 인사이트

이번 업데이트에서 흥미로운 점은 `heritageradio` 프로젝트의 폭발적인 활동량입니다. 30일 기준 897건의 커밋이 기록되었습니다. 20년 넘게 엔지니어 생활을 해오며 느낀 점은, 숙련된 개발자일수록 '큰 덩어리의 커밋'보다는 '작고 빈번한 커밋'을 선호한다는 것입니다.

작은 단위의 커밋은 다음과 같은 이점을 줍니다.
- **리스크 감소:** 문제가 생겼을 때 롤백해야 할 범위가 명확합니다.
- **리뷰 효율성:** 동료(혹은 미래의 나)가 코드를 이해하는 속도가 빨라집니다.
- **심리적 성취감:** 작은 마일스톤을 계속 달성하며 개발 동력을 유지할 수 있습니다.

### 마치며

씨피랩스에서 CTO로 근무하며 수많은 팀을 리딩하고 기술적 의사결정을 내리지만, 결국 기본은 '꾸준한 기록'과 '작은 개선의 반복'에 있다고 믿습니다. 

여러분의 대시보드에는 어떤 기록들이 쌓이고 있나요? 단순히 `fixed bug`라는 메시지보다, 어떤 고민을 통해 그 버그를 잡았는지 기록하는 습관을 가져보시길 권합니다. 그것이 훗날 여러분의 가장 강력한 기술적 자산이 될 것입니다.
