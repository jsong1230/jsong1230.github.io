안녕하세요, 시니어 소프트웨어 엔지니어이자 코드 리뷰어입니다.

오늘 제출하신 커밋은 단순한 기능 구현을 넘어, **배포 파이프라인의 신뢰성(Reliability)을 확보하기 위한 검증 단계 추가**와 **인프라 설정 정리**가 핵심으로 보입니다. 특히 S3 ETag의 특성을 이용해 데이터 무결성을 검증하려는 접근 방식이 매우 인상적입니다.

상세 리뷰 결과입니다.

---

## ✅ 잘한 점

1. **데이터 무결성 검증 로직 도입 (`verify_prod_sync.py`)**
   - 단순히 "파일 전송 성공"이라는 API 응답에 의존하지 않고, `dev archive (md5) ↔ prod S3 (ETag)`를 비교하는 **Byte-identical 검증**을 추가한 점이 매우 훌륭합니다.
   - 특히 파일 사이즈가 작아 S3의 Single-part upload 특성(ETag == MD5)을 정확히 파악하고 이를 검증 로직에 활용한 기술적 판단이 매우 날카롭습니다.

2. **Fail-safe 메커니즘 설계**
   - 검증 실패 시 `invalidation skip` $\rightarrow$ `Telegram 알림` $\rightarrow$ `hc_ping /fail` $\rightarrow$ `exit 1`으로 이어지는 흐름을 통해, 잘못된 데이터가 캐시 갱신을 통해 사용자에게 전파되는 것을 원천 차단했습니다. 이는 운영 환경에서 발생할 수 있는 최악의 시나리오를 방지하는 시니어 수준의 방어적 설계입니다.

3. **운영 가시성 확보**
   - 텔레그램 알림 시 단순히 "실패했다"가 아니라, `tail -10`을 통해 구체적인 에러 로그의 일부를 함께 전송하도록 하여, 운영자가 즉시 원인을 파악하고 대응할 수 있게 배려한 점이 좋습니다.

4. **인프라 잔재 정리 (`compute-stack.ts`)**
   - `HLS_BUILD_PENDING_DISABLED=1` 설정을 통해 사용하지 않는 beat schedule 잔재를 정리함으로써, 인프라 설정의 단순함을 유지하고 잠재적인 사이드 이펙트를 제거했습니다.

## 🔧 개선할 점

1. **셸 스크립트 내 코드 중복 (DRY 원칙)**
   - `sync_today_cycle1_weekend.sh`, `sync_today_cycle5_to_prod.sh`, `sync_tomorrow_cycle_to_prod.sh` 세 파일에서 `verify_prod_sync.py`를 호출하고 에러를 처리하는 로직이 거의 동일하게 중복되어 있습니다.
   - **개선 제안:** 검증 및 알림 로직을 별도의 셸 함수나 스크립트(예: `scripts/verify_and_notify.sh`)로 분리하여 호출하는 방식을 권장합니다. 유지보수 시 세 군데를 모두 수정해야 하는 번거로움을 줄일 수 있습니다.

2. **임시 파일 관리의 안정성**
   - `VERIFY_OUT=$(mktemp)`를 통해 임시 파일을 생성하고 있으나, 스크립트 중간에 예기치 못한 오류로 종료될 경우 `rm -f "$VERIFY_OUT"`이 실행되지 않아 `/tmp` 디렉토리에 찌꺼기 파일이 남을 수 있습니다.
   - **개선 제안:** 셸의 `trap` 명령어를 사용하여 스크립트 종료 시(EXIT) 무조건 임시 파일이 삭제되도록 설정하는 것이 더 안전합니다.
     ```bash
     VERIFY_OUT=$(mktemp)
     trap 'rm -f "$VERIFY_OUT"' EXIT
     ```

3. **하드코딩된 경로**
   - `/home/jsong/heritageradio/scripts/verify_prod_sync.py`와 같이 절대 경로가 하드코딩되어 있습니다. 이는 실행 환경(사용자 계정이나 디렉토리 구조)이 변경될 때 스크립트가 깨지는 원인이 됩니다.
   - **개선 제안:** `${SCRIPT_DIR}` 변수를 정의하거나, 스크립트 위치를 기준으로 상대 경로를 계산하여 호출하도록 수정하십시오.

## 💡 오늘의 기술 포인트

**"S3 ETag를 활용한 무결성 검증 패턴"**
오늘 작업의 핵심은 S3의 ETag가 Single-part upload 시 파일의 MD5 해시값과 동일하다는 점을 이용한 것입니다. 이는 별도의 메타데이터 저장소 없이도 S3 API 호출만으로 원본 파일과 업로드된 파일의 일치 여부를 빠르게 확인할 수 있는 매우 효율적인 패턴입니다. 

다만, 추후 파일 크기가 커져 **Multi-part upload**가 발생하면 ETag 계산 방식이 달라지므로(각 파트의 해시를 다시 해싱), 파일 크기 임계치를 모니터링하거나 Multi-part 대응 로직을 고려해야 한다는 점을 팀원들과 공유하면 좋을 것 같습니다.
