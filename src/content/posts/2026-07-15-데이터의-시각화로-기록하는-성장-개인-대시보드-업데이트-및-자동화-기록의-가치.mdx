# [Dev Log] 데이터의 흐름으로 기록하는 나의 성장 궤적: 대시보드 업데이트

안녕하세요, 씨피랩스의 CTO 송주한(Jeffrey)입니다.

엔지니어에게 있어 '기록'은 단순한 메모 이상의 의미를 갖습니다. 특히 20년 넘게 개발과 아키텍처 설계라는 길을 걸어오며 제가 깨달은 점은, 파편화된 커밋 로그보다 **'시각화된 흐름'**이 주는 인사이트가 훨씬 강력하다는 것입니다. 

오늘은 제 개인 대시보드의 데이터를 업데이트하며, 제가 왜 이런 자동화된 기록 시스템을 유지하고 있는지, 그리고 오늘 변경된 데이터가 시사하는 바가 무엇인지 공유해보려 합니다.

### 🛠️ 오늘의 작업: 데이터 동기화와 상태 업데이트

오늘 진행한 작업은 단순해 보이지만 제 일상의 기술적 리듬을 반영하는 `dashboard.json` 파일의 업데이트였습니다. 

```json
// src/data/dashboard.json 변경 사항 중 일부
- "last_msg": "[admin] briefing 재응답: exit1 진짜원인=FK(sequence아님) + 근본 A방�",
+ "last_msg": "[main] docs: A-3 briefing — usePlayTracker fix staged (차기 릴리스 대기",
```

단순히 날짜와 메시지를 바꾸는 작업처럼 보이지만, 이 diff에는 제가 최근 집중하고 있는 문제들이 고스란히 담겨 있습니다. `heritageradio` 레포지토리의 변경 사항을 보면, 단순한 버그 수정을 넘어 `usePlayTracker`와 같은 상태 관리 로직의 최적화와 릴리스 전략을 고민한 흔적이 보입니다.

### 💡 기술적 결정과 인사이트

저는 제 활동량을 `commits_7d`(최근 7일 커밋 수)와 `commits_30d`(최근 30일 커밋 수)라는 지표로 관리합니다. 이번 업데이트에서 특히 흥미로운 부분은 다음과 같습니다.

1. **활동 밀도의 변화**: `heritageradio` 프로젝트의 7일 커밋 수가 35개에서 47개로 급증했습니다. 이는 특정 기능의 구현 단계에서 집중적인 리팩토링과 테스트가 이루어졌음을 의미합니다.
2. **관심사의 확장**: `mini-jsong` 머신에서 기존의 `metadium-explorer-stack` 대신 `health`라는 새로운 레포지토리가 업데이트되었습니다. 기술적인 성취만큼이나 중요한 것이 '지속 가능한 건강 관리'라는 관점에서, 이를 데이터화하여 추적하는 시스템을 구축했습니다.
3. **실전적 문제 해결**: `mirae` 레포지토리의 `adr_track` 스크립트 수정 건(`SKHYV stale 종가 물림 → 정식 SKHY 우선 조회`)은 실제 금융 데이터의 특성(Stale data)을 처리하는 엣지 케이스를 해결한 사례입니다.

### 🚀 마치며: 엔지니어의 '대시보드'가 필요한 이유

많은 분이 "왜 굳이 JSON 파일로 이런 데이터를 관리하시나요?"라고 묻습니다. 

씨피랩스에서 수많은 프로젝트를 리딩하며 느낀 점은, 복잡한 시스템일수록 **'현재 어디에 리소스가 집중되고 있는가'**를 객관적으로 보는 것이 중요하다는 것입니다. 저 역시 개인의 성장과 학습 곡선을 데이터로 확인하며, 제가 어느 분야에 매너리즘을 느끼고 있는지, 혹은 어디에 과도하게 몰입하고 있는지를 체크합니다.

코드 한 줄의 변경보다 더 중요한 것은 그 변경이 향하는 **방향성**입니다. 내일의 저는 오늘의 데이터보다 조금 더 성장해 있기를 기대하며, 오늘의 로그를 마칩니다.
