# [Dev Log] 데이터의 흐름으로 기록하는 성장: 대시보드 업데이트와 자동화의 가치

안녕하세요, 씨피랩스의 CTO 송주한(Jeffrey)입니다.

엔지니어에게 있어 '기록'은 단순히 과거를 남기는 것이 아니라, 자신의 성장 궤적을 데이터로 확인하는 과정입니다. 저는 개인적인 생산성 관리와 학습 흐름을 파악하기 위해 깃허브 활동 데이터를 시각화한 개인 대시보드를 운영하고 있습니다. 오늘은 이 대시보드의 데이터를 업데이트하며 느낀 몇 가지 인사이트를 공유해보려 합니다.

### 단순한 업데이트, 그 이상의 의미

오늘 진행한 작업은 겉보기에 매우 단순합니다. `src/data/dashboard.json` 파일의 수치를 최신화하는 `chore` 작업이었죠. 

```json
- "last_msg": "Auto-generate daily posts",
+ "last_msg": "Auto-generate weekly book post",
...
- "commits_30d": 68
+ "commits_30d": 69
```

하지만 이 작은 diff 안에는 제가 지향하는 **'자동화된 기록 시스템'**의 철학이 담겨 있습니다. 매일 혹은 매주 제가 어떤 저장소에서 어떤 고민을 했는지, 커밋 메시지는 무엇이었는지를 수동으로 적는 것은 매우 소모적인 일입니다. 대신, 스크립트를 통해 데이터를 추출하고 JSON 형태로 관리함으로써 저는 '기록하는 행위' 자체에 드는 에너지를 줄이고, '기록된 데이터를 분석하는 것'에 더 집중할 수 있게 되었습니다.

### 기술적 결정: 왜 JSON 기반의 정적 데이터인가?

실시간 API를 통해 대시보드를 구현할 수도 있었겠지만, 저는 의도적으로 JSON 파일에 데이터를 저장하고 커밋하는 방식을 선택했습니다. 여기에는 두 가지 이유가 있습니다.

1. **불변성의 기록 (Immutability):** API는 현재의 상태만을 보여주지만, Git에 커밋된 JSON 파일은 특정 시점의 내 상태를 스냅샷처럼 보존합니다. 시간이 흐른 뒤 "내가 2026년 7월에는 어떤 프로젝트에 몰입했지?"라는 질문에 가장 정확한 답을 줄 수 있는 방법입니다.
2. **성능과 단순함:** 복잡한 백엔드 서버 없이 정적 페이지(GitHub Pages)만으로도 충분히 빠른 응답 속도를 확보할 수 있습니다.

### 인사이트: 데이터가 말해주는 우선순위

오늘 업데이트된 데이터를 살펴보면 `heritageradio`나 `metadium_homepage` 등 다양한 프로젝트의 활동량이 보입니다. 특히 `commits_30d` 수치의 변화를 통해 제가 최근 어디에 더 많은 리소스를 투입하고 있는지 객관적으로 파악할 수 있었습니다.

엔지니어는 때로 '열심히 하고 있다'는 느낌(Feeling)에 매몰되기 쉽습니다. 하지만 이렇게 수치화된 데이터(Fact)를 마주하면, 현재 나의 리소스 배분이 계획과 일치하는지, 혹은 특정 프로젝트에 과하게 매몰되어 다른 중요한 일을 놓치고 있지는 않은지 냉정하게 판단할 수 있습니다.

### 마치며

씨피랩스에서 수많은 아키텍처를 설계하고 팀을 리딩하며 제가 배운 가장 큰 교훈 중 하나는 **"측정할 수 없다면 개선할 수 없다"**는 것입니다. 이는 거대한 시스템 설계뿐만 아니라 개인의 성장 관리에도 동일하게 적용됩니다.

오늘의 작은 커밋 하나가 모여 저의 기술적 자산이 되고, 그것이 다시 더 나은 의사결정의 근거가 됩니다. 여러분도 자신만의 '성장 대시보드'를 만들어보시는 건 어떨까요?
