# [DevLog] 데이터의 흐름으로 기록하는 나의 성장: 대시보드 자동화와 회고

안녕하세요, 씨피랩스의 CTO 송주한(Jeffrey Joo-Han Song)입니다.

엔지니어에게 있어 '기록'은 단순한 메모 이상의 의미를 갖습니다. 제가 20년 넘게 개발자로 활동하며 깨달은 점은, 내가 어떤 고민을 했고 어떤 리듬으로 작업했는지를 객관적인 데이터로 바라볼 때 비로소 진정한 성장이 보인다는 것입니다. 

오늘은 제 개인 대시보드의 데이터를 업데이트하며, 단순한 `chore` 작업 너머에 있는 '데이터 기반의 자기 객관화'에 대해 이야기해보려 합니다.

### 🛠 오늘의 작업: 대시보드 데이터 갱신

오늘 진행한 작업은 제 GitHub 활동 데이터를 수집해 대시보드에 반영하는 `dashboard.json` 파일의 업데이트였습니다. 겉으로 보기에는 단순한 JSON 값의 변경이지만, 이 데이터가 생성되는 과정과 그 결과값이 주는 인사이트가 핵심입니다.

```json
// src/data/dashboard.json 변경 사항 중 일부
{
  "name": "jsong1230.github.io",
  "last_date": "2026-07-04",
  "last_msg": "Auto-generate weekly book post",
  "branch": "main",
  "activity": [2, 2, 2, 2, 2, 3, 0],
  "commits_7d": 16,
  "commits_30d": 69
}
```

### 💡 기술적 결정과 인사이트

**1. 자동화된 파이프라인의 가치**
이번 업데이트에서 흥미로운 점은 `last_msg`에 나타난 `"Auto-generate weekly book post"`라는 문구입니다. 저는 단순히 데이터를 수동으로 입력하는 것이 아니라, 특정 주기마다 책 리뷰나 학습 내용을 자동으로 생성하고 이를 대시보드에 반영하는 파이프라인을 구축해 두었습니다. 

Web3나 AI 분야처럼 변화 속도가 극심한 도메인에서 일하는 엔지니어에게 가장 무서운 것은 '정체'입니다. 이를 방지하기 위해 강제적으로라도 학습하고 기록하는 시스템을 자동화함으로써, 최소한의 성실함을 시스템적으로 보장받는 전략을 택한 것입니다.

**2. 활동 지표(Activity)를 통한 리듬 분석**
`activity: [2, 2, 2, 2, 2, 3, 0]`와 같은 배열 데이터는 저의 한 주간 작업 리듬을 보여줍니다. 매일 꾸준히 커밋을 이어가다 주말에 휴식하는 패턴이 명확히 드러나죠. 씨피랩스에서 수많은 프로젝트를 리딩하며 느낀 점은, 단기적인 스퍼트보다 '지속 가능한 속도(Sustainable Pace)'를 유지하는 것이 장기적으로 훨씬 높은 퀄리티의 결과물을 만든다는 것입니다.

### 🚀 마치며

단순한 JSON 파일의 업데이트였지만, 이를 통해 제가 지난 30일간 69번의 커밋을 하며 꾸준히 기록해왔음을 다시 한번 확인했습니다. 

많은 분이 최신 프레임워크나 복잡한 아키텍처에 집중하지만, 결국 시니어 엔지니어의 내공은 **'자신만의 루틴을 어떻게 시스템화하고, 그 데이터를 어떻게 해석하여 개선하느냐'**에서 결정된다고 생각합니다.

여러분은 자신의 성장을 어떤 데이터로 증명하고 계신가요? 때로는 복잡한 코드보다, 정직하게 쌓인 커밋 로그 한 줄이 더 큰 확신을 주곤 합니다.
