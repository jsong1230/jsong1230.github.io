# [Dev Log] 데이터의 흐름으로 기록하는 나의 성장 궤적

안녕하세요, 씨피랩스(CP Labs)의 CTO 송주한입니다.

엔지니어로서 20년 넘게 커리어를 쌓아오며 제가 가장 중요하게 생각하는 것 중 하나는 '측정 가능한 성장'입니다. 단순히 "열심히 했다"는 느낌보다, 내가 어떤 리듬으로 코드를 작성하고, 어떤 프로젝트에 에너지를 쏟고 있는지를 객관적인 데이터로 확인하는 과정이 성장에 큰 도움이 되기 때문입니다.

오늘은 제 개인 대시보드의 데이터를 업데이트하는 작업을 진행했습니다. 단순한 `chore` 작업처럼 보일 수 있지만, 이 데이터가 갖는 의미와 제가 이를 관리하는 방식에 대해 이야기해보려 합니다.

### 데이터가 말해주는 현재의 집중도

이번 커밋의 핵심은 `src/data/dashboard.json` 파일의 갱신입니다. 특히 눈에 띄는 부분은 `heritageradio` 저장소의 활동 지표입니다.

```json
- "commits_7d": 18,
- "commits_30d": 47
+ "commits_7d": 32,
+ "commits_30d": 64
```

최근 7일간의 커밋 수가 18개에서 32개로 급증한 것을 볼 수 있습니다. 이는 최근 제가 집중하고 있는 설계 작업과 로드맵 정리가 실제 구현 단계로 빠르게 전이되고 있음을 보여줍니다. 특히 `last_msg`에 기록된 *"docs: roadmap·test-pending stale 정리"*라는 문구는, 단순히 코드를 짜는 것을 넘어 프로젝트의 부채(stale)를 걷어내고 소프트 런칭을 위한 기반을 다지는 과정에 있음을 시사합니다.

### 기술적 결정: 정적 데이터 기반의 대시보드

사실 이런 데이터는 API를 통해 실시간으로 가져오는 것이 일반적입니다. 하지만 저는 일부러 `dashboard.json`이라는 정적 파일에 데이터를 기록하고 커밋하는 방식을 택했습니다. 여기에는 두 가지 이유가 있습니다.

1. **히스토리의 영속성**: API 호출 방식은 '현재 상태'만 보여주지만, Git 커밋으로 관리하면 시간이 흐른 뒤 특정 시점에 내가 어떤 리듬으로 일했는지 `git diff`를 통해 복기할 수 있습니다.
2. **오버헤드 최소화**: 개인 대시보드에 복잡한 백엔드 인프라를 구축하는 것은 과잉 엔지니어링(Over-engineering)입니다. 데이터 파일 하나를 업데이트하는 것만으로도 충분한 가시성을 확보할 수 있습니다.

### 인사이트: 리듬을 관리하는 엔지니어

코드 변경 사항 중 `activity` 배열의 숫자 변화를 보면, 어떤 날은 폭발적으로 작업하고 어떤 날은 정적인 흐름을 유지하는 저만의 '개발 리듬'이 보입니다. 

많은 시니어 엔지니어들이 범하는 실수 중 하나가 '매일 동일한 양의 생산성'을 강요하는 것입니다. 하지만 창의적인 설계와 복잡한 버그 수정은 선형적으로 일어나지 않습니다. 때로는 깊은 고민의 시간(0의 기록)이 필요하고, 그 응축된 시간이 터져 나올 때 폭발적인 커밋(17, 14 등의 기록)이 발생합니다.

씨피랩스에서 수많은 팀을 리딩하며 배운 점 역시, 팀원 개개인의 이러한 리듬을 존중하고 전체적인 흐름(Velocity)을 관리하는 것이 훨씬 효율적이라는 사실입니다.

### 마치며

오늘의 작업은 단순한 숫자 업데이트였지만, 저에게는 지난 일주일의 몰입도를 확인하고 다음 단계의 에너지를 설정하는 리프레시의 시간이었습니다. 

여러분은 자신의 성장을 어떻게 측정하고 계신가요? 때로는 복잡한 툴보다, 나만의 단순한 기록 방식이 가장 강력한 피드백 루프가 되기도 합니다.
