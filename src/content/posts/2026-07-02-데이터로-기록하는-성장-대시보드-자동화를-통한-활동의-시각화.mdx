# [Dev Log] 데이터의 흐름으로 기록하는 성장: 대시보드 자동화와 '활동의 시각화'

안녕하세요, 씨피랩스(CP Labs)의 CTO 송주한입니다.

엔지니어로서 20년 넘게 커리어를 쌓아오며 제가 가장 중요하게 생각하는 것 중 하나는 **'기록의 자동화'**입니다. 단순히 무엇을 했느냐를 적는 일기를 넘어, 나의 기술적 활동이 어떤 궤적을 그리며 움직이고 있는지를 데이터로 확인하는 과정은 메타인지를 높이는 데 매우 효과적이기 때문입니다.

오늘은 제 개인 프로젝트인 `gram-jsong`의 대시보드 데이터를 업데이트하며, 제가 추구하는 '데이터 기반의 자기 관리'에 대해 이야기해보려 합니다.

### 오늘의 작업: 대시보드 데이터 최신화

오늘 진행한 작업은 단순해 보이지만, 제 개발 생태계를 한눈에 보여주는 `dashboard.json` 파일을 업데이트하는 것이었습니다. 

```json
// src/data/dashboard.json 변경 부분
{
  "generated_at": "2026-07-01T23:00:01.309Z",
  "machines": [
    {
      "name": "gram-jsong",
      "repos": [
        {
          "name": "jsong1230.github.io",
          "last_msg": "Auto-generate daily posts",
          "activity": [2, 3, 3, 2, 2, 2, 0],
          "commits_7d": 16,
          "commits_30d": 68
        },
        ...
```

이번 커밋에서 주목할 점은 단순히 날짜를 바꾼 것이 아니라, 최근 7일 및 30일간의 커밋 수(`commits_7d`, `commits_30d`)와 일별 활동량(`activity`)을 갱신했다는 점입니다. 특히 `jsong1230.github.io` 저장소의 경우, 최근 30일간 68개의 커밋이 발생하며 꾸준한 기록의 흐름을 보여주고 있습니다.

### 기술적 결정과 인사이트: 왜 '데이터'인가?

많은 개발자가 GitHub의 잔디(Contribution Graph)를 통해 자신의 성실함을 증명하곤 합니다. 하지만 저는 여기서 한 걸음 더 나아가, **내가 어떤 저장소에, 어떤 빈도로, 어떤 메시지를 남기며 시간을 쓰고 있는지를 구조화된 데이터(JSON)로 관리**하는 방식을 택했습니다.

이렇게 데이터를 구조화하면 다음과 같은 이점이 있습니다.

1. **추세 분석**: 특정 기간에 특정 프로젝트(`claude-agent` 등)에 몰입했는지, 혹은 여러 프로젝트에 분산되어 있었는지를 수치로 파악할 수 있습니다.
2. **자동화의 선순환**: `Auto-generate daily posts`라는 커밋 메시지에서 알 수 있듯이, 이 데이터 수집 과정 자체가 자동화되어 있습니다. 기록을 위한 기록에 시간을 쓰는 것이 아니라, 시스템이 기록하게 하고 저는 그 결과물을 통해 인사이트를 얻는 구조입니다.
3. **객관적 회고**: "최근에 열심히 한 것 같다"라는 느낌이 아니라, "지난 7일간 16번의 유의미한 변경이 있었다"라는 팩트를 기반으로 회고할 수 있습니다.

### 마치며

씨피랩스에서 CTO로서 복잡한 Web3 인프라와 AI 에이전트 설계를 리딩하다 보면, 때로는 거대한 시스템의 흐름에 매몰되기 쉽습니다. 그럴 때일수록 이렇게 개인의 작은 활동 단위(Atomic unit)를 데이터로 기록하고 시각화하는 습관이 중심을 잡는 데 큰 도움이 됩니다.

결국 훌륭한 엔지니어링이란 복잡한 문제를 단순하게 만드는 것이며, 나 자신의 성장 과정 또한 하나의 '최적화된 시스템'으로 만드는 과정이라고 생각합니다.

오늘 여러분의 대시보드에는 어떤 데이터가 쌓였나요? 기록되지 않은 노력은 잊히기 쉽지만, 데이터로 남은 노력은 반드시 성장의 자산이 됩니다.
