# [Dev Log] 데이터의 '길이'와 '맥락' 사이의 균형 잡기: 오디오 스크립트 Trim 전략

안녕하세요, 씨피랩스의 CTO 송주한입니다.

엔지니어로서 시스템을 설계하다 보면, 기술적인 제약 조건(Constraint)과 사용자 경험(UX) 사이에서 타협점을 찾아야 하는 순간이 많습니다. 오늘은 제가 진행 중인 프로젝트 중 하나인 `heritageradio`에서 발생한 '콘텐츠 길이 초과' 문제를 해결하며 느낀 인사이트를 공유해 보려 합니다.

### 1. 문제의 정의: "10분을 넘기면 안 된다"

오디오 콘텐츠 서비스에서 '시간'은 매우 엄격한 제약 조건입니다. 특정 슬롯에 맞춰 송출되어야 하는 콘텐츠가 10분을 초과하면, 전체 편성표가 밀리거나 강제로 끊기는 불상사가 발생합니다. 

확인 결과, 약 25편의 에피소드가 600초(10분)를 초과하고 있었습니다. 단순히 뒤를 싹둑 자르는 방식은 가능하겠지만, 그렇게 되면 콘텐츠의 완성도를 결정짓는 '맺음말(Farewell)'이 사라져 청취자에게 불쾌한 경험을 줄 수 있습니다.

### 2. 기술적 결정: 단순 절단이 아닌 '맥락 보존형 Trim'

저는 단순히 글자 수를 맞추는 것이 아니라, **콘텐츠의 구조(Structure)를 이해하는 Trim 알고리즘**을 구현하기로 했습니다. 

가장 핵심이 된 로직은 `trim_script` 함수입니다. 저는 스크립트를 단락(`\n\n`) 단위로 분리한 뒤, 다음과 같은 우선순위를 적용했습니다.

1. **인트로(First Paragraph)와 맺음말(Last Paragraph)은 무조건 보존한다.**
2. **중간 단락들을 누적하며 타겟 길이(590초 분량)에 도달할 때까지만 포함시킨다.**
3. **만약 단락이 너무 적어 구조적 분리가 어렵다면, 마침표(`.`) 단위를 찾아 자연스럽게 마무리한다.**

실제 구현된 코드의 일부를 살펴보면 이 고민이 드러납니다.

```python
# 마지막 단락 (farewell) 보존
last_para = paragraphs[-1]
middle_paras = paragraphs[1:-1]  # 중간
first_para = paragraphs[0]  # 인트로

# ... (중략) ...

# 중간 단락 처음부터 누적, target 이하로
available = target_len - base_len
kept_middle = []
cur_len = 0
for p in middle_paras:
    seg_len = len(p) + 2  # \n\n
    if cur_len + seg_len > available:
        break
    kept_middle.append(p)
    cur_len += seg_len
```

단순히 `script[:target_len]`으로 자르는 대신, `middle_paras`를 루프 돌며 가용한 길이(`available`)만큼만 채우는 방식을 택했습니다. 이렇게 하면 콘텐츠의 시작과 끝이라는 '맥락'은 유지하면서, 중간의 상세 내용을 조절해 전체 길이를 맞출 수 있습니다.

### 3. 배운 점과 인사이트: 자동화와 정밀함의 조화

이번 작업에서 흥미로웠던 점은 **'자동화 도구의 정교함'**입니다. 25편의 데이터를 일일이 수정하는 것은 비효율적이지만, 그렇다고 무분별한 자동화는 퀄리티 저하를 야기합니다. 

결국 개발자의 역할은 **"도메인의 특성(오디오 스크립트의 구조)을 코드로 어떻게 치환하느냐"**에 있다는 것을 다시 한번 느꼈습니다. 또한, `dry-run` 옵션을 통해 실제 DB에 반영하기 전 대상 리스트와 결과물을 미리 검증하는 프로세스를 갖추는 것이 운영 단계에서의 사고를 막는 가장 확실한 방법임을 체감했습니다.

현재 이 작업으로 재합성(Regen)이 시작되었으며, 약 3.7시간 뒤면 최적화된 오디오 콘텐츠들이 준비될 예정입니다. 

기술은 결국 사람에게 닿는 서비스의 일부입니다. 1초의 오차를 줄이는 정밀함과, 한 문장의 맥락을 살리는 세심함이 만날 때 진정한 사용자 경험이 완성된다고 믿습니다.
