# [Dev Log] 엣지 케이스의 무서움: 오디오 큐 동기화와 슬롯 침범 문제 해결기

안녕하세요, 씨피랩스의 CTO 송주한(Jeffrey Joo-Han Song)입니다.

엔지니어로서 가장 당혹스러운 순간은 "내 로컬에서는 잘 되는데, 특정 환경의 실사용자에게서만 재현되는 간헐적 버그"를 마주했을 때입니다. 오늘 제가 겪은 일이 딱 그랬습니다. 모바일 앱에서 백그라운드 재생 중 다음 프로그램으로 넘어가지 않고 멈춰버리는 현상이 보고되었습니다.

단순한 네트워크 지연인 줄 알았으나, 파고 들어보니 **비동기 상태 관리의 허점**과 **콘텐츠 길이의 불확실성**이 결합된 복합적인 문제였습니다.

### 1. 문제의 핵심: 왜 멈췄는가? (Root Cause)

현상은 간단했습니다. S10 기기에서 앱을 백그라운드로 돌린 지 10분 정도 지나면 다음 슬롯의 프로그램이 시작되지 않고, 화면을 켜면 그제야 작동하는 것이었습니다. 분석 결과 세 가지 지점에서 문제가 발생하고 있었습니다.

**첫째, 비동기 Fetch와 큐 생성의 레이스 컨디션**
다음 슬롯의 곡 리스트(`nextSlotPieces`)를 가져오는 작업은 비동기로 이루어집니다. 그런데 이 데이터가 도착하기도 전에 재생 로더가 `setSrcList`를 호출해버리면, 큐에는 현재 슬롯의 곡들만 담기게 됩니다. 결과적으로 큐가 바닥나면서 재생이 정지되는 사고가 발생한 것입니다.

**둘째, 너무 가혹했던 'LOCKOUT' 정책**
이전 코드에서는 큐가 부족한 상태에서 스킵을 시도하면 이를 에러로 간주하고 24시간 동안 재시도하지 않는 `LOCKOUT` 상태로 진입하게 설계되어 있었습니다. 큐 부족이라는 일시적 상태가 '영구적 정지'라는 최악의 결과로 이어진 셈입니다.

**셋째, 슬롯 시간을 초과하는 'Tail Music'**
나레이션 뒤에 붙는 배경음악(Tail Music)이 예상보다 길어지면서 정해진 슬롯 시간을 침범하고, 이것이 전체 스케줄링의 꼬임으로 이어졌습니다.

### 2. 기술적 결정과 해결책

저는 이 문제를 해결하기 위해 **백엔드-서비스-UI** 전 계층에서 3중 방어선을 구축했습니다.

#### A. 상태 게이트 도입 (App.tsx)
단순히 데이터의 유무(`null` 체크)만으로는 '데이터를 가져오는 중'인지 '데이터가 진짜 없는 것'인지 구분할 수 없었습니다. 그래서 `nextSlotReady`라는 명시적인 상태 값을 추가했습니다.

```typescript
// v0.9.53: fetch 완료 여부를 명시적으로 추적
const [nextSlotReady, setNextSlotReady] = useState(false);
```
이제 로더는 `nextSlotReady`가 `true`가 된 후에만 큐를 설정합니다. 이를 통해 큐가 항상 `[현재 슬롯 곡들, 다음 슬롯 곡들]`의 형태를 유지하도록 보장했습니다.

#### B. 유연한 재시도 전략 (service.ts)
큐 부족 시 무조건 LOCKOUT에 걸던 로직을 수정했습니다. 큐가 부족하면 일단 0.5초 후 재시도하도록 하여, `wake-resync`를 통해 데이터가 채워진 후 자연스럽게 재생이 이어지도록 변경했습니다.

#### C. 절대 상한선(max_sec) 강제 (backend/music_slot.py)
백엔드 API에 `max_sec` 파라미터를 추가하여, 슬롯 시간을 절대 넘지 않는 곡만 선택하도록 필터링을 강화했습니다.

```python
if max_sec is not None:
    candidates = [
        c for c in candidates
        if c["duration_sec"] and c["duration_sec"] <= max_sec
    ]
    if not candidates:
        raise HTTPException(409, f"음악 풀에 max_sec={max_sec} 이하 곡 없음")
```
이제 클라이언트는 `(슬롯 전체 시간 - 나레이션 길이 - 시그널 길이 - 마진)`을 계산해 `max_sec`를 보냅니다. "적당히 긴 곡"이 아니라 "절대로 이 시간을 넘지 않는 곡"을 가져오게 함으로써 물리적인 슬롯 침범을 원천 차단했습니다.

### 3. 마치며: 엔지니어의 인사이트

이번 작업에서 다시 한번 느낀 점은 **"분산된 시스템에서 '완료'의 정의를 명확히 해야 한다"**는 것입니다. 단순히 변수가 `null`이 아니라고 해서 작업이 완료된 것이 아니라, 명시적인 `ready` 상태를 통해 상태 전이를 관리하는 것이 얼마나 중요한지 깨달았습니다.

또한, 사용자 경험(UX) 측면에서는 어르신들이 많이 사용하시는 서비스 특성을 고려해 편성표 모달의 폰트 크기를 키우는 등 가독성 개선 작업도 병행했습니다. 기술적인 정교함만큼이나 중요한 것이 결국 '사용자가 어떻게 느끼는가'라는 기본임을 다시금 상기한 하루였습니다.

내일은 더 안정적인 서비스로 찾아뵙겠습니다.
