# [Dev Log] 데이터의 흐름으로 기록하는 개발자의 성장 곡선

안녕하세요, 씨피랩스(CP Labs)의 CTO 송주한입니다.

개발자에게 있어 '기록'은 단순한 메모 이상의 의미를 갖습니다. 내가 어떤 고민을 했고, 어떤 기술적 결정을 내렸으며, 시간이 흐르며 어떻게 성장했는지를 보여주는 가장 정직한 지표가 되기 때문입니다. 저는 개인적으로 저의 개발 활동량을 시각화하여 보여주는 대시보드를 운영하고 있는데, 오늘 그 데이터를 업데이트하며 몇 가지 생각할 점이 있어 정리해 보았습니다.

### 단순한 데이터 업데이트, 그 이상의 의미

오늘 진행한 작업은 표면적으로는 `dashboard.json` 파일의 데이터를 갱신하는 단순한 `chore` 작업이었습니다. 하지만 이 JSON 파일에 담긴 수치들은 제가 최근 집중하고 있는 기술적 관심사와 업무의 우선순위를 그대로 투영하고 있습니다.

```json
// src/data/dashboard.json 변경 사항 중 일부
{
  "name": "heritageradio",
  "last_msg": "[admin+main] CHANGELOG: episode-registry flat 전파 history + confirmed_at tz f",
  "commits_7d": 21,
  "commits_30d": 51
}
```

특히 `heritageradio` 프로젝트의 커밋 메시지를 보면, 최근 '데이터 구조의 평탄화(flat)'와 '타임존(tz) 처리'라는 구체적인 문제 해결에 집중했음을 알 수 있습니다. 분산 시스템이나 Web3 환경에서 데이터의 일관성을 유지하고 시간축을 맞추는 작업은 매우 까다롭지만, 서비스의 안정성을 결정짓는 핵심적인 디테일입니다.

### 기술적 결정과 인사이트: '가시성'의 힘

저는 왜 굳이 이런 대시보드 데이터를 수동 혹은 반자동으로 관리할까요? 20년 넘게 엔지니어로 살아오며 느낀 점은, **"측정되지 않는 것은 개선될 수 없다"**는 것입니다.

1. **자기 객관화**: 특정 프로젝트에 과하게 몰입하고 있지는 않은지, 혹은 방치되고 있는 레포지토리는 없는지 한눈에 파악할 수 있습니다.
2. **맥락의 복원**: `last_msg`를 통해 과거의 내가 어떤 고민을 했는지 빠르게 리마인드하며 컨텍스트 스위칭 비용을 줄입니다.
3. **동기 부여**: `commits_7d`와 같은 수치적 변화는 정체되어 있지 않고 계속해서 무언가를 만들어내고 있다는 심리적 만족감을 줍니다.

### 마치며

오늘의 커밋은 비록 코드 한 줄의 로직을 바꾼 것은 아니지만, 저의 개발 궤적을 기록하고 관리하는 중요한 루틴의 일부였습니다. 씨피랩스에서도 팀원들이 각자의 성장을 가시적으로 확인하고, 서로의 기술적 도전 과제를 공유할 수 있는 문화를 만드는 데 관심이 많습니다.

결국 훌륭한 소프트웨어는 훌륭한 엔지니어의 집요한 기록과 개선 의지에서 나온다고 믿습니다. 내일은 또 어떤 기록들이 저의 대시보드를 채우게 될지 기대됩니다.
