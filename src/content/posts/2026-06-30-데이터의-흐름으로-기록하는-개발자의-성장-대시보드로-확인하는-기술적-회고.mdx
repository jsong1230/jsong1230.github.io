## [Dev Log] 데이터의 흐름으로 기록하는 개발자의 성장 곡선

안녕하세요, 씨피랩스의 CTO 송주한(Jeffrey)입니다.

개발자에게 있어 '기록'은 단순히 과거를 남기는 행위가 아니라, 현재의 내가 어디에 있고 어느 방향으로 성장하고 있는지를 확인하는 이정표와 같습니다. 저는 개인적인 생산성 관리와 기술적 회고를 위해 저만의 대시보드를 운영하고 있는데, 오늘은 그 대시보드의 데이터를 업데이트하며 느낀 몇 가지 생각들을 공유해보려 합니다.

### 단순한 데이터 업데이트, 그 이상의 의미

오늘 진행한 작업은 표면적으로는 `dashboard.json` 파일의 수치를 변경하는 단순한 `chore` 작업이었습니다. 하지만 이 JSON 파일 하나에는 제가 최근 집중하고 있는 프로젝트들의 활동량과 기술적 관심사가 고스란히 담겨 있습니다.

이번 업데이트에서 가장 눈에 띄는 부분은 `metadium_homepage` 저장소의 변화입니다.

```json
{
  "name": "metadium_homepage",
  "last_date": "2026-06-29",
  "last_msg": "배포 인프라(Docker/Caddy용) + SEO 추가 (챗봇 제외)",
  "branch": "master",
  "activity": [0, 0, 0, 0, 0, 23, 0],
  "commits_7d": 39,
  "commits_30d": 39
}
```

이전 기록에서는 단순히 '잔재 정리' 수준의 작업이었다면, 이번에는 **Docker와 Caddy를 활용한 배포 인프라 구축과 SEO 최적화**라는 실질적인 엔지니어링 작업이 집중적으로 이루어졌습니다. 특히 `activity` 배열의 특정 일자에 '23'이라는 숫자가 찍힌 것을 보며, 인프라 설정 단계에서 겪었던 치열한 트러블슈팅의 시간이 떠오릅니다.

### 기술적 결정: 왜 인프라와 SEO에 집중했는가?

Web3와 DID 같은 고도의 기술적 도메인일수록, 그 가치를 사용자에게 전달하는 '접점(Interface)'의 안정성과 접근성이 중요합니다. 아무리 훌륭한 블록체인 기술이라도 사용자가 찾지 못하거나(SEO 문제), 접속했을 때 불안정하다면(인프라 문제) 그 가치는 반감됩니다.

따라서 이번 작업에서는 다음과 같은 결정들을 내렸습니다.
1. **Caddy 도입**: 설정이 간결하면서도 자동 HTTPS 인증을 지원하는 Caddy를 선택해 배포 파이프라인의 복잡도를 낮췄습니다.
2. **SEO 최적화**: 검색 엔진이 Web3 서비스의 정체성을 정확히 파악할 수 있도록 메타데이터 구조를 정교화했습니다.

### 인사이트: 정량적 데이터가 주는 정성적 피드백

많은 개발자가 커밋 수나 활동 그래프(Contribution Graph)에 집착하는 경향이 있습니다. 하지만 저는 이 데이터를 '성과'가 아닌 '패턴'으로 읽으려 노력합니다.

오늘의 `diff`를 보면 `heritageradio`나 `claude-agent` 같은 다른 프로젝트들의 수치는 낮아졌지만, 특정 프로젝트의 수치가 급증했습니다. 이는 제가 현재 **'선택과 집중'**을 통해 리소스를 효율적으로 배분하고 있다는 정량적인 증거가 됩니다. 

20년 넘게 엔지니어로 일하며 배운 점 중 하나는, 모든 것을 다 잘하려는 욕심보다 **지금 이 순간 가장 임팩트가 큰 지점이 어디인지 찾아내고 그곳에 몰입하는 능력**이 훨씬 중요하다는 것입니다.

### 마치며

단순한 JSON 파일의 숫자 변경이었지만, 이를 통해 최근의 작업 흐름을 복기하고 다음 단계의 우선순위를 설정할 수 있었습니다. 여러분도 자신만의 방식으로 '성장의 궤적'을 기록하고 계신가요? 

데이터는 거짓말을 하지 않습니다. 다만 그 데이터를 어떻게 해석하고 어떤 인사이트를 끌어내느냐가 진정한 시니어 엔지니어의 역량이라고 믿습니다.

내일은 더 깊이 있는 기술 이야기로 돌아오겠습니다.
