# [Dev Log] 작은 디테일이 만드는 안정성: CI/CD 업데이트와 빌드 에러 해결기

안녕하세요, 씨피랩스(CP Labs)의 CTO 송주한(Jeffrey Joo-Han Song)입니다.

엔지니어로서 가장 보람찬 순간은 거대한 아키텍처를 설계할 때이기도 하지만, 때로는 시스템의 구석구석을 살펴 '잠재적인 위험'을 제거하고 효율을 높이는 디테일한 작업에서 오기도 합니다. 오늘은 제 개인 블로그의 자동화 파이프라인을 정비하며 겪은 몇 가지 기술적 결정과 해결 과정을 공유해보려 합니다.

### 1. 선제적 대응: Node 24 업그레이드

가장 먼저 진행한 작업은 GitHub Actions 워크플로우의 런타임을 Node 20에서 Node 24로 업그레이드한 것입니다. 

```yaml
- name: Setup Node
  uses: actions/setup-node@v5
  with:
    node-version: '24'
    cache: 'npm'
```

단순한 버전 숫자 변경처럼 보이지만, 이는 **'기술적 부채의 선제적 관리'**라는 관점에서 중요합니다. GitHub의 지원 종료(Deprecation) 일정이 다가오기 전에 미리 최신 메이저 버전으로 전환함으로써, 갑작스러운 빌드 실패라는 리스크를 제거했습니다. 20년 넘게 엔지니어링을 하며 배운 점 중 하나는, 인프라 업데이트를 미루면 결국 가장 바쁜 시점에 가장 큰 장애로 돌아온다는 것입니다.

### 2. 엣지 케이스의 발견: `.mdx.mdx`와 앵글 브래킷(`<>`)

자동화 스크립트를 운영하다 보면 예상치 못한 '엣지 케이스'를 만나게 됩니다. 이번에는 두 가지 재미있는 문제가 있었습니다.

첫 번째는 파일 확장자가 중복으로 붙는 `.mdx.mdx` 현상이었습니다. 파일명을 생성하는 `createRecapMDX()` 함수에서 확장자 처리 로직이 중복 적용되었더군요. 16개의 파일이 잘못 생성된 것을 확인하고, 로직 수정과 함께 기존 파일들의 이름을 일괄 변경했습니다.

두 번째는 더 흥미로운 **MDX 빌드 실패** 건이었습니다. 주간 회고(Weekly Recap) 콘텐츠 내에 앵글 브래킷(`<`, `>`)이 포함되면서 MDX 파서가 이를 HTML 태그나 JSX 컴포넌트로 오인해 빌드 에러를 일으킨 것입니다.

이를 해결하기 위해 `scripts/generate-weekly-recap.js`에서 특수 문자를 적절히 이스케이프(Escape) 처리하거나 텍스트로 안전하게 렌더링하도록 로직을 보완했습니다. 데이터가 코드로 변환되는 지점에서 발생하는 전형적인 '파싱 에러'였지만, 자동화 파이프라인에서는 이런 작은 문자 하나가 전체 배포를 중단시킬 수 있다는 점을 다시금 상기했습니다.

### 3. 인사이트: 자동화의 역설

오늘 작업을 통해 느낀 점은 **"자동화가 고도화될수록, 그 자동화를 관리하는 '관리 포인트' 역시 정교해져야 한다"**는 것입니다. 

단순히 스크립트를 짜서 돌리는 것을 넘어, 
- 런타임 버전의 생명주기를 관리하고(Node 24 업그레이드),
- 입력 데이터의 무결성을 검증하며(확장자 중복 제거),
- 파싱 단계의 예외 상황을 처리하는(MDX 특수문자 해결) 

과정들이 뒷받침되어야 진정한 의미의 '자동화'가 완성됩니다. 씨피랩스에서 복잡한 Web3 및 AI 시스템을 구축할 때도 마찬가지입니다. 화려한 기능보다 더 중요한 것은 어떤 상황에서도 무너지지 않는 견고한 파이프라인과 예외 처리의 디테일입니다.

오늘의 작은 수정들이 모여 더 안정적인 기록의 공간이 되길 바랍니다. 여러분의 파이프라인은 안녕한가요?
