# [Dev Log] 엣지 케이스의 연속: 시간 랩핑, 날짜 경계, 그리고 체이닝 로직 해결기

안녕하세요, 씨피랩스의 CTO 송주한(Jeffrey Joo-Han Song)입니다.

개발자에게 가장 즐거우면서도 괴로운 순간은 언제일까요? 저는 아마 "분명히 로직상으로는 맞는데, 실제 사용자 환경에서 예상치 못한 값이 튀어나올 때"가 아닐까 합니다. 오늘 진행한 모바일 앱(v0.9.27~v0.9.29) 업데이트 작업이 딱 그랬습니다. 

단순해 보이지만 놓치기 쉬운 세 가지 엣지 케이스를 해결하며 얻은 인사이트를 공유합니다.

---

### 1. "마이너스 시간"의 습격: 모듈로 연산의 함정
가장 먼저 마주한 문제는 시간 표시 포맷팅이었습니다. KST 0:00 기준으로 오프셋이 적용되면서 `blockStart` 값이 음수로 떨어지는 경우가 발생했습니다. 사용자에게 **"-1:30~00:00 별처럼 빛나는 노래"**라고 표시되는 당혹스러운 상황이었죠.

프로그래밍 언어마다 다르지만, 많은 언어에서 `%`(modulo) 연산자는 피연산자가 음수일 때 음수를 반환합니다. 이를 해결하기 위해 흔히 사용하는 '더블 모듈로' 패턴을 적용했습니다.

```typescript
// fix: ((n % 1440) + 1440) % 1440 wrap
```

단순히 `n % 1440`을 하는 것이 아니라, 한 바퀴(1440분)를 더해준 뒤 다시 모듈로 연산을 수행함으로써 어떤 음수 값이 들어와도 항상 0~1439 사이의 양수 범위로 랩핑(wrapping)되도록 수정했습니다. 기초적인 수학적 처리지만, UX 관점에서는 매우 치명적인 버그였기에 다시 한번 기본기의 중요성을 느꼈습니다.

### 2. 날짜 경계(Day Boundary)와 캐시 전략
두 번째는 자정 직후의 데이터 처리 문제입니다. 5월 2일 0시 1분이 되었을 때, 사용자는 여전히 '어제의 마지막 프로그램'을 보고 있어야 합니다. 하지만 단순하게 '오늘'의 타임테이블만 가져온다면, 자정 직후에 갑자기 프로그램이 바뀌거나 끊기는 현상이 발생합니다.

이를 위해 **`yesterdayTimetable`** 상태를 추가하고, 날짜 경계 검출 로직을 구현했습니다.
- `safeIdx === 0` 이고 `cycleNum === 0`이며 어제 데이터가 존재할 때 $\rightarrow$ 어제의 마지막 슬롯을 표시.

결과적으로 5월 2일 0시 직후에도 **"23:50~00:00 약이 되는 밥상"**이라는 어제(평일)의 마지막 슬롯이 정확하게 표시되는 것을 확인했습니다.

### 3. 오진(Misdiagnosis)과 데이터 구조의 재발견
가장 흥미로웠던 점은 '주말 재방송' 기능의 버그 수정 과정이었습니다. 처음에는 백엔드에서 데이터를 제대로 내려주지 않아 음악만 재생된다고 판단해 백엔드 수정 계획까지 세웠습니다. 하지만 정밀 진단 결과, 백엔드는 이미 `episode_sequence`를 통해 5일치 데이터를 정상적으로 응답하고 있었습니다.

**진짜 문제는 모바일 클라이언트의 판단 로직에 있었습니다.** 
기존 로직은 `episode.audio_path`가 없으면 바로 음악 슬롯으로 폴백(fallback)하도록 설계되어 있었습니다. 하지만 주말의 `voice_50` 슬롯은 단일 파일이 아니라 5개의 에피소드 체인으로 구성되어 있었죠.

```typescript
// fix: slot_kind === "voice_50" && episode_sequence?.length 분기 추가
```

단일 오디오 경로만 체크하던 로직에 `episode_sequence` 존재 여부를 확인하는 분기를 추가함으로써, 백엔드 수정 없이 클라이언트 수정만으로 5일치 보이스 체인이 정상적으로 재생되도록 해결했습니다.

### 💡 마치며: 엔지니어로서의 인사이트

오늘 작업을 통해 다시금 깨달은 점은 **"가설을 세우고 검증하는 단계에서 성급한 결론을 내리지 말 것"**입니다. 백엔드 문제라고 확신하고 수정 계획을 세웠지만, 실제 데이터(Payload)를 다시 뜯어보니 클라이언트의 해석 오류였던 것처럼 말이죠.

특히 Web3나 AI처럼 복잡한 데이터 흐름을 다루는 환경일수록, 로그와 실제 응답 값을 기반으로 한 '데이터 중심의 사고'가 디버깅 시간을 획기적으로 줄여줍니다. 

오늘도 엣지 케이스와 싸우며 한 뼘 더 성장한 하루였습니다. 내일은 더 견고한 코드로 찾아뵙겠습니다.
