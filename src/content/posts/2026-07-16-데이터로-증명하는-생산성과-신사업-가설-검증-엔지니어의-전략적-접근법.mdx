# [Dev Log] 데이터의 흐름을 잡고, 신사업의 가설을 검증하는 법

안녕하세요, 씨피랩스(CP Labs)의 CTO 송주한입니다.

엔지니어로서의 삶은 단순히 코드를 짜는 것에 그치지 않습니다. 때로는 자동화된 대시보드를 통해 나의 생산성을 추적하고, 때로는 비즈니스 모델의 기술적 실현 가능성을 빠르게 검증하는 '아키텍트'이자 '전략가'의 역할이 더 중요할 때가 많습니다.

오늘은 제가 진행한 두 가지 성격의 작업—**개인 생산성 대시보드 업데이트**와 **신사업 MVP를 위한 딥리서치**—를 통해 느낀 인사이트를 공유해보려 합니다.

---

### 1. 데이터로 증명하는 루틴: Dashboard Update

먼저 개인 저장소인 `jsong1230.github.io`에서 대시보드 데이터를 업데이트했습니다. 저는 제가 관리하는 여러 레포지토리의 커밋 활동량을 JSON 형태로 관리하여 시각화하고 있습니다.

이번 변경 사항 중 흥미로운 점은 레포지토리 명칭의 변경입니다.
```diff
- "name": "metadium_homepage",
+ "name": "cplabs_homepage",
```
단순한 이름 변경처럼 보이지만, 이는 제가 소속된 **씨피랩스**의 정체성을 프로젝트에 반영하는 과정입니다. 기술적인 관점에서 보면, 정적인 JSON 데이터를 통해 나의 활동 지표(`commits_7d`, `commits_30d`)를 추적하는 것은 일종의 '자기 객관화' 과정입니다. 어떤 프로젝트에 리소스가 쏠려 있는지, 혹은 방치되고 있는 영역은 없는지 한눈에 파악함으로써 다음 주의 우선순위를 결정하는 근거로 활용합니다.

### 2. 가설을 빠르게 부수는 기술: 신사업 MVP 딥리서치

오늘의 메인 작업은 `claude-agent`를 활용한 신사업 A(혜택 플랫폼)의 MVP 스펙 구체화였습니다. CTO로서 제가 가장 경계하는 것은 **"기술적으로 가능할 것 같다"라는 막연한 추측으로 개발을 시작하는 것**입니다.

이번 리서치에서 제가 집중한 것은 '제약 사항의 조기 발견'이었습니다. 딥리서치 워크플로우를 통해 106개의 에이전트와 24개의 소스를 분석한 결과, 다음과 같은 치명적인 제약들을 발견했습니다.

*   **공공 API의 한계**: "숨은 환급금 자동조회" 기능이 핵심 가치라고 생각했지만, 실제로는 관련 오픈 API가 없어 구조적으로 불가능함을 확인했습니다.
*   **법적/정책적 장벽**: 카카오톡 친구톡의 '5만 친구 장벽'과 정보통신망법 제50조라는 구체적인 제약을 확인했습니다.

**여기서 내린 기술적/전략적 결정은 '피벗(Pivot)'입니다.**
불가능한 '자동 조회'에 매달리는 대신 '찾는 법 안내'로 기능을 변경하고, 통신/렌탈 CPA의 정면 승부보다는 지원금 API와 앱테크 영역으로 진입 경로를 수정했습니다.

```markdown
- MVP 스펙 v0.2 3대 변경: 
  ① 통신·렌탈 CPA 정면 회피 → 지원금(무료 API)·앱테크·쿠팡파트너스 진입 
  ② "숨은 환급금 자동조회" 보류 → "찾는 법 안내" 
  ③ 카톡 5만친구 장벽 → 콘텐츠+웹+수신동의 병행
```

### 💡 마치며: 엔지니어의 인사이트

오늘 작업을 통해 다시 한번 느낀 점은 **"최고의 코드는 짜지 않아도 되는 코드"**라는 것입니다. 딥리서치를 통해 불가능한 기능을 미리 걷어냄으로써, 수주 간의 헛된 개발 공수를 줄일 수 있었습니다.

또한, 모든 산출물을 HTML로 저장하는 규칙을 세운 것은 나중에 이 결정 과정(Decision Log)을 쉽게 추적하고 아카이빙하기 위함입니다. 20년 넘게 엔지니어로 일하며 깨달은 것은, 결국 성공하는 제품은 '화려한 기술'이 아니라 '정확한 가설 검증'과 '빠른 실행'에서 온다는 사실입니다.

내일은 오늘 확인한 실사 항목(CPA 실단가 및 마이데이터 환급조회)을 바탕으로 더 정교한 스펙을 다듬어보겠습니다.
