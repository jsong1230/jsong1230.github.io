# [Dev Log] 사용자 경험의 디테일이 제품의 완성도를 결정한다: 접근성과 심사 대응기

안녕하세요, 씨피랩스의 CTO 송주한입니다.

최근 진행 중인 'Heritage Radio' 프로젝트의 MVP 런칭을 앞두고, 어제 심야부터 오늘 새벽까지 꽤 밀도 높은 작업을 진행했습니다. 단순히 '기능이 돌아가게 만드는 것'을 넘어, 실제 사용자가 제품을 접했을 때 느끼는 '완성도'와 앱 스토어라는 '관문'을 통과하기 위한 디테일한 보강 작업이었습니다.

오늘은 제가 어떤 고민을 했고, 이를 기술적으로 어떻게 풀었는지 공유해 보려 합니다.

---

### 1. 시간대 중립화: AI 콘텐츠의 범용성 확보
가장 먼저 해결한 문제는 AI가 생성하는 스크립트의 '시간적 제약'이었습니다. 기존에는 생성 시점의 시간대(아침, 저녁 등)를 프롬프트에 넣었는데, 이를 그대로 두면 낮에 듣는 방송에서 "편안한 저녁 되세요"라는 인사가 나올 수 있는 문제가 있었습니다.

이를 해결하기 위해 `TIME_NEUTRAL_RULE`이라는 공통 규칙을 도입했습니다. "하루 4시간 블록이 6회 반복 재생된다"는 전제를 프롬프트에 명시하고, 시간대별 인사를 금지하는 지시어를 삽입했습니다. 

**인사이트:** AI 서비스에서 '컨텍스트'는 양날의 검입니다. 너무 구체적인 컨텍스트는 특정 상황에서 오류(Hallucination이나 부적절한 톤앤매너)를 유발합니다. 범용적인 재생 모델을 구축할 때는 의도적으로 컨텍스트를 '중립화'하는 설계가 필요함을 다시금 느꼈습니다.

### 2. '진짜' 앱다운 경험: 접근성과 햅틱 피드백
웹 뷰 기반의 하이브리드 앱일수록 사용자는 "웹사이트를 보고 있다"는 느낌을 받기 쉽습니다. 저는 이 간극을 줄이기 위해 **접근성(A11y)**과 **햅틱(Haptics)**에 집중했습니다.

*   **접근성 보강:** 어르신들이 주 타겟인 서비스인 만큼, `aria-live="polite"`를 통해 상태 변화를 스크린리더가 자연스럽게 읽어주도록 설정하고, `data-text-size` 속성을 이용해 CSS 변수(`--text-scale`)로 글자 크기를 유연하게 조절하는 시스템을 구축했습니다.
*   **햅틱 피드백:** `@capacitor/haptics`를 도입하여 다이얼을 돌리거나 버튼을 누를 때 미세한 진동이 느껴지도록 구현했습니다. 

```typescript
// lib/haptics.ts (개념적 구현)
// 네이티브 햅틱을 우선하되, 지원하지 않는 환경에서는 navigator.vibrate로 폴백
// 30ms 쓰로틀링을 통해 과도한 진동 방지
```

단순히 버튼 하나를 누르는 행위에 '촉각적 피드백'을 더하는 것만으로도 앱의 신뢰도와 사용성이 비약적으로 상승합니다. 엔지니어로서 코드의 효율성만큼이나 중요한 것이 이러한 '감성적 디테일'이라는 점을 매번 깨닫습니다.

### 3. 심사 통과를 위한 전략적 문서화
마지막으로 Apple과 Google의 까다로운 앱 심사를 대비해 `app-review-notes.md`를 작성했습니다. 특히 Apple의 '최소 기능 요구사항(4.2 Minimum Functionality)' 가이드라인에 대응하기 위해, 이 앱이 왜 단순한 웹 래퍼가 아닌 '앱 전용 기능(백그라운드 오디오, 로컬 알림, 햅틱 등)'을 갖춘 서비스인지를 논리적으로 정리했습니다.

### 마치며: 구현보다 중요한 것은 '검증'
작업을 마치며 `test-pending.md`라는 미검증 항목 트래커를 새로 만들었습니다. 코드를 짰다고 해서 끝난 것이 아니라, **"실제 경로당에서 어르신들이 사용했을 때, 24시간 연속 재생 시 롤오버가 정상적으로 일어나는가?"**와 같은 실환경 검증이 남았기 때문입니다.

기술적 구현은 수단일 뿐, 결국 정답은 사용자의 손끝에 있습니다. 씨피랩스에서의 모든 개발 여정도 늘 이 '실제 경험'에 초점을 맞추려 노력하고 있습니다.

내일은 실제 기기 4종과 파일럿 테스트를 통해 이 가설들을 검증해 보겠습니다.
