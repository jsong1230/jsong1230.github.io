## [Dev Log] 데이터로 기록하는 성장: 개인 대시보드 업데이트와 '지속성'의 가치

안녕하세요, 씨피랩스의 CTO 송주한(Jeffrey)입니다.

엔지니어에게 있어 '기록'은 단순히 과거를 남기는 행위가 아니라, 자신의 성장 궤적을 추적하고 병목 지점을 찾아내는 중요한 피드백 루프입니다. 저는 개인적으로 제가 관리하는 여러 프로젝트의 활동량을 시각화한 대시보드를 운영하고 있는데, 오늘 그 데이터를 최신화하며 몇 가지 생각할 거리가 생겨 공유해보려 합니다.

### 🛠️ 오늘의 작업: 대시보드 데이터 파이프라인 업데이트

오늘 진행한 작업은 간단해 보이지만 제 개발 루틴의 핵심인 `dashboard.json` 데이터를 업데이트하는 것이었습니다. 

```json
// src/data/dashboard.json 변경 부분
+ {
+   "name": "thepol-dev",
+   "last_date": "2026-05-18",
+   "last_msg": "Initial thepol dev infra (docker-compose + nginx + patches + README)",
+   "branch": "main",
+   "activity": [0, 0, 0, 0, 0, 1, 0],
+   "commits_7d": 1,
+   "commits_30d": 1
+ }
```

이번 업데이트에서 가장 흥미로운 점은 새로운 프로젝트인 `thepol-dev`가 대시보드에 추가되었다는 것입니다. Docker-compose와 Nginx를 활용한 인프라 초기 설정을 마쳤고, 이제 막 첫 발을 뗐습니다. 

또한, 기존에 진행하던 `heritageradio` 프로젝트의 경우 `episode-registry 누적 주차` 관련 업데이트가 반영되었습니다. 단순한 기능 구현을 넘어, 데이터의 누적 방식과 사용자 경험(UX)을 개선하는 디테일한 작업들이 계속되고 있음을 알 수 있습니다.

### 💡 기술적 결정과 인사이트: '정량적 지표'가 주는 힘

많은 개발자가 "코드만 잘 짜면 되지, 굳이 커밋 횟수나 활동량을 기록할 필요가 있을까?"라고 묻곤 합니다. 하지만 20년 넘게 엔지니어링 현장에서 CTO로 활동하며 제가 배운 점은, **측정할 수 없는 것은 개선할 수 없다**는 사실입니다.

이번 diff에서 보듯 `commits_7d`나 `activity` 배열(일주일간의 일별 커밋 수)을 관리하는 이유는 단순히 '열심히 했다'를 증명하기 위함이 아닙니다.

1. **리듬 파악**: 특정 요일에 작업량이 몰리거나 급감하는 패턴을 보며, 제 컨디션과 집중도가 어느 시점에 최상인지 파악합니다.
2. **관성 유지**: `activity` 배열에 `0`이 많아지기 시작하면, 이는 기술적 슬럼프나 매너리즘의 신호일 수 있습니다. 이때 의도적으로 작은 작업이라도 수행하며 '개발 관성'을 유지합니다.
3. **맥락 복구**: `last_msg`를 통해 내가 최근에 어떤 고민을 하며 코드를 수정했는지 빠르게 리마인드할 수 있습니다.

### 🚀 마치며

씨피랩스에서 복잡한 블록체인 인프라와 Web3 서비스, AI 통합 솔루션을 설계할 때도 이 원칙은 동일하게 적용됩니다. 거대한 시스템일수록 작은 단위의 정량적 지표들이 모여 전체의 건강 상태(Health Check)를 나타내기 때문입니다.

오늘의 업데이트는 단순한 JSON 파일 수정이었지만, 저에게는 새로운 프로젝트의 시작을 알리는 신호탄이자 지난 일주일의 몰입도를 확인하는 성찰의 시간이었습니다. 여러분은 자신의 성장을 어떻게 기록하고 계신가요?

기록하는 습관이 결국 더 나은 아키텍처를 만드는 밑거름이 된다고 믿습니다. 내일은 또 어떤 기록이 쌓일지 기대하며 오늘 글을 마칩니다.
